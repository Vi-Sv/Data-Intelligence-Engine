# Кейс №19: CTE-магистрали и изоляция аномальных проводок

## Блок: Высшая Лига SQL / Управление рисками
**Заказчик:** Финансовый директор / Департамент комплаенса.

### Бизнес-задача (ТЗ):
> *"Для предотвращения нарушений и выявления крупных финансовых аномалий необходимо разработать отказоустойчивый и легкочитаемый инструмент мониторинга транзакционного потока. Требуется сформировать аналитический реестр с выводом нормализованных имен контрагентов и чистой суммы совершенных ими операций (нейтрализовать пропуски, перевести отрицательные маркеры возвратов в абсолютную величину валового оборота). На итоговую витрину необходимо вывести исключительно те аномальные проводки, где чистая сумма операции строго превысила порог в 300 000 рублей. Результаты упорядочить по убыванию объема транзакций."*

---

## Исходная структура СУБД (Таблица `agent_transactions`):
*   `tx_id` (INT) — Уникальный номер финансовой транзакции (Первичный ключ).
*   `agent_name` (VARCHAR) — Юридическое наименование торгового агента / контрагента.
*   `amount` (INT) — Сумма операции в рублях (содержит системные пропуски `NULL` и отрицательные маркеры возвратов средств).

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического скрипта была применена методология декомпозиции логики через обобщенные табличные выражения (CTE). Это изолирует этапы предварительной нормализации данных от финальной фильтрации, повышая общую читаемость кода и упрощая его дальнейшую поддержку на Highload-проектах крупных корпораций.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Инициализация обобщенного выражения (`WITH`):** До выполнения основного запроса движок СУБД разворачивает изолированное виртуальное пространство под техническим именем `clean_data` в оперативной памяти сервера.
2. **Обеспечение качества данных (Data Quality):** Внутри изолированного цеха скалярный комбо-резец `TRIM(LOWER())` очищает текстовые поля контрагентов от пробелов и регистровых ошибок. Связка функций `ABS(COALESCE())` локализует пропуски данных нулями и принудительно трансформирует отрицательные маркеры возвратов холдинга в чистый положительный валовой оборот.
3. **Обращение к виртуальной магистрали:** Главный внешний запрос обращается напрямую к сформированной стерильной временной таблице `clean_data` как к физическому справочнику базы данных (в Python Pandas данная логика реализована через создание изолированной копии фрейма).
4. **Хирургическое сито верхнего уровня (`WHERE`):** Фильтр верхнего этажа `WHERE clean_data.g2 > 300000` выполняет функцию контрольно-пропускного пункта. Он отсекает фоновый операционный шум, оставляя на финальной витрине только аномальные крупные транзакции, строго превышающие установленный лимит.
5. **Вывод витрины:** Итоговый реестр упорядочивается по убыванию финансового веса сумм посредством оператора `ORDER BY DESC`.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и верификации CTE-конвейера данных были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за зазора моторики):** Падение парсера базы данных на этапе компиляции подзапроса из-за пропущенной парной закрывающей круглой скобки на входе в скалярную комбинацию функций `ABS(COALESCE(amount,0)`. Движок выдавал ошибку нарушения синтаксического контура.
    *   *Метод устранения:* Структура скобок верифицирована, парный закрывающий символ интегрирован на уровне ядра подзапроса.
*   **Риск №2 (Рыночный эталон — логический капкан нестрогого порога):** Ошибка фильтрации вследствие ошибочного применения нестрогого оператора `>=` вместо строгого `>`. Код захватывал в отчет транзакции, равные ровно 300 000 единиц, что нарушало регламентное ТЗ финансового директора по выявлению чистых аномалий.
    *   *Метод устранения:* Нестрогое условие принудительно заменено в черновике на строгий знак превосходства (`>`), обеспечив абсолютную стерильность и точность выводимой витрины.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная магистральная витрина комплаенс-контроля интегрируется с BI-платформы (Apache Superset / Yandex DataLens) для непрерывного мониторинга транзакционных рисков.

### Структура дашборда:
1. **Реестр аномальной активности (Risk Leaderboard):** Интерактивная таблица «Контрагент -> Сумма транзакции», автоматически выводящая операции, прошедшие жесткий финансовый фильтр, отсортированная по убыванию, для мгновенной локализации пиковых нагрузок.
2. **Индикатор совокупного объема рисков (KPI Card):** Крупный цифровой виджет, отображающий общую сумму замороженных или требующих дополнительной верификации средств по аномальным проводкам (`SUM(amount)`).
3. **Модуль исторического сопоставления контрагентов:** Фильтр по именам агентов, позволяющий сотрудникам безопасности моментально изучать историю транзакций конкретного партнера и выявлять системность совершения аномально крупных платежей.

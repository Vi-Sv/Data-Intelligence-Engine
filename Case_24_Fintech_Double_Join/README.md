# Кейс №24: CTE-магистрали и сквозная сшивка двойного LEFT JOIN

## Блок: Финтех / Финансовый аудит и комплаенс
**Заказчик:** Руководитель отдела комплаенса / Финансовый директор.

### Бизнес-задача (ТЗ):
> *"Для проведения сверки внутренних балансов и выявления транзакционных закономерностей необходимо разработать сквозной аналитический алгоритм консолидации данных. Требуется объединить три независимых источника: справочник клиентов, лог фактических майских платежей и реестр начисленных бонусных баллов. На итоговую витрину необходимо вывести нормализованные имена контрагентов, у которых суммарный объем проведенных оплат абсолютно эквивалентен величине начисленных бонусов. Важнейшее техническое требование: клиенты без операций ('молчуны') должны быть гарантированно сохранены в контуре вычислений на этапе сшивки, а системные пропуски заменены на числовой ноль. Результаты отсортировать по алфавиту."*

---

## Исходная структура СУБД (Сопряжение таблиц один-ко-многим-ко-многим):

### 1. Справочник клиентов (`clients`):
*   `client_id` (INT) — Уникальный номер клиента в системе (Первичный ключ).
*   `client_name` (VARCHAR) — ФИО / Наименование контрагента.

### 2. Реестр платежей (`client_payments`):
*   `pay_id` (INT) — Уникальный номер транзакции оплаты (Первичный ключ).
*   `client_id` (INT) — Идентификатор клиента (Внешний ключ связи).
*   `pay_amount` (INT) — Фактическая сумма перевода в рублях.

### 3. Лог бонусных начислений (`client_bonuses`):
*   `bonus_id` (INT) — Уникальный номер операции начисления (Первичный ключ).
*   `client_id` (INT) — Идентификатор клиента (Внешний ключ связи).
*   `bonus_amount` (INT) — Сумма начисленных бонусов.

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры сквозного аналитического куба была применена модель изоляции сопряжений внутри обобщенного табличного выражения (CTE). Это позволяет выполнить последовательную сквозную сборку реляционного моста из трех таблиц до этапа финальной фильтрации строк, гарантируя консистентность метрик и высокую производительность вычислительного ядра на Highload-нагрузках.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Инициализация откомпилированной магистрали (`WITH compiled_balances`):** СУБД разворачивает виртуальное пространство в оперативной памяти (ОЗУ) до старта основного тела запроса. Скалярный резец `TRIM(LOWER())` выбривает текстовые поля имен клиентов от хаотичного регистрового шума и пробелов ручного ввода.
2. **Развертывание двойного левостороннего моста:** Исходный неприкасаемый справочник `clients` (`t1`) сопрягается со второй таблицей лога оплат (`t2`) через первый оператор `LEFT JOIN`. Полученный транзитный массив последовательно объединяется с третьей таблицей бонусов (`t3`) через второй оператор `LEFT JOIN`. Критически важно, что сопряжение третьего моста замкнуто строго на родительский ключ `t1.client_id`, а не на таблицу оплат `t2`, что намертво блокирует риск аннигиляции (автоматического стирания) клиентов-молчунов, у которых отсутствовали платежи.
3. **Обеспечение качества данных (Data Quality):** Функции `COALESCE` перехватывают образующиеся на стыке пустых связей `NULL` и переводят их в чистые числовые нули, формируя стерильную базу для верификации.
4. **Хирургическое сито финального отбора (`WHERE`):** Главный внешний запрос вызывает откомпилированную витрину. Фильтр `WHERE compiled_balances.g3 = compiled_balances.g4` осуществляет сквозную сверку балансов, пропуская на экран строго строки с абсолютным равенством оплат и бонусов. Лишние внешние алиасы удалены для сохранения бритвенной читаемости.
5. **Вывод витрины:** Финальный реестр выстраивается по алфавитному порядку имен клиентов с помощью `ORDER BY ASC`.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и верификации многомерного конвейера данных были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за зазора моторики — лишняя запятая):** Падение компилятора базы данных на этапе парсинга из-за разделительного символа запятой (`,`), ошибочно влепленного после объявления базовой таблицы `t1` непосредственно перед оператором моста (`FROM clients AS t1, LEFT JOIN`). Запятая принудительно переключала движок на логику Cross Join, что вызывало жесткий конфликт синтаксиса.
    *   *Метод устранения:* Лишний символ удален, структура реляционного сопряжения приведена к промышленному стандарту.
*   **Риск №2 (Логический капкан сшивания второго LEFT JOIN — потеря молчунов):** Критическая логическая ошибка черновика, при которой мост таблицы бонусов `t3` был ошибочно привязан к ключу таблицы оплат `t2.client_id` вместо справочника `t1.client_id`. При отсутствии у клиента платежей в мае таблица `t2` генерировала пустоту `NULL`, связь с бонусами рвалась, и клиент полностью стирался из финальной выборки, уничтожая саму суть левостороннего объединения.
    *   *Метод устранения:* Проведен архитектурный надзор, контур сопряжения жестко перепривязан к нерушимому справочнику-донору `t1.client_id`.
*   **Риск №3 (Коллизия ложного вызова алиаса на верхнем этаже):** Падение компиляции в главном запросе из-за попытки вызвать поля по старому длинному имени CTE-цеха (`compiled_balances.g3`) при наличии объявленного короткого внешнего алиаса `AS FSTT`. СУБД выдавала отсечку: «Таблица compiled_balances не существует», так как старое имя стиралось новым маркером.
    *   *Метод устранения:* Избыточный внешний алиас `FSTT` полностью удален из структуры для сохранения бритвенной читаемости и стабильности кода.
*   **Риск №4 (Механическая опечатка наименования колонки):** Локальный заусенец моторики с дублированием текстового маркера на витрине (`"Клиенты_чей_платеж_равен_сумме_платежа"`).
    *   *Метод устранения:* Текст скорректирован в строгом соответствии с регламентным ТЗ: слово «платеж» заменено на «бонус».

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная сквозная витрина финтех-комплаенса интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для непрерывной верификации балансов.

### Структура дашборда:
1. **Консолидированный реестр эквивалентных балансов (Audited Users):** Интерактивная таблица, автоматически выводящая список контрагентов, прошедших жесткое сито фильтрации равенства, отсортированная по алфавиту, для оперативного предоставления отчетности регуляторам.
2. **Индикатор транзакционной синхронности (KPI Card):** Цифровой информер, отображающий общее количество верифицированных пользователей, у которых начисления совпали со списаниями.
3. **Панель сквозного текстового поиска:** Интерактивное поле ввода для мгновенного поиска конкретного клиента по ФИО и проверки статуса его баланса на витрине.

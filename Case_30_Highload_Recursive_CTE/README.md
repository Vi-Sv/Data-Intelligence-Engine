# Кейс №30: Многоступенчатые рекурсивные CTE и построение графовых иерархий штата

## Блок: Высшая Лига SQL / Архитектура сложных структур данных (R&D)
**Статус:** Техническая документация повышенной сложности (Исследовательский полигон).
**Заказчик:** Директор по организационному развитию / HR-директор холдинга.

### Бизнес-задача (ТЗ):
> *"Для проведения масштабного организационного аудита компании, анализа уровней подчиненности и выявления избыточных управленческих звеньев необходимо спроектировать динамическую древовидную модель штатной структуры холдинга. Требуется на основе таблицы штатного расписания сформировать сквозной реестр с указанием ID сотрудника, его нормализованного имени и ID его непосредственного руководителя. В смежном четвертом столбце необходимо рассчитать абсолютный уровень вложенности персонала внутри корпоративного графа (где Генеральный директор занимает базовый уровень №1, а линейные сотрудники — конечные ветви дерева). Вывод упорядочить по возрастанию уровней подчиненности."*

---

## Исходная структура СУБД (Таблица `company_structure`):
*   `emp_id` (INT) — Уникальный номер сотрудника (Первичный ключ).
*   `emp_name` (VARCHAR) — ФИО сотрудника (содержит регистровый шум).
*   `manager_id` (INT) — Идентификатор непосредственного руководителя (Внешний ключ связи типа «Self-Join» на эту же таблицу; для Генерального директора содержит системную пустоту `NULL`).

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического скрипта была развернута продвинутая рекурсивная модель СУБД ANSI SQL — `WITH RECURSIVE`. Данный паттерн незаменим при обработке нелинейных сетевых структур и направленных графов (сетевые графики, древовидные справочники, оргструктуры), где глубина вложенности элементов заранее неизвестна. 

### Пошаговый алгоритм выполнения рекурсивного запроса:
1. **Развертывание анкерного члена (База рекурсии):** Движок СУБД инициализирует первое изолированное пространство в ОЗУ. Выполняется стартовый отбор записей, у которых `manager_id IS NULL`, что фиксирует корневой элемент графа (Генеральный директор). Ему присваивается константа `level = 1`, а строковое имя нормализуется комбо-резцами `TRIM(LOWER())`.
2. **Запуск рекурсивного итератора (`UNION ALL`):** СУБД открывает каскадный шаг рекурсии. Внутреннее вычислительное ядро начинает последовательно вызывать само себя (`FROM hierarchy_tree AS t1`), сопрягая строки внешней грязной таблицы `company_structure` (`t2`) с результатами предыдущего шага по рекурсивному мосту связи `ON t2.manager_id = t1.g1`. На каждом шагу спуска по ветке вниз глубина узла автоматически инкрементируется (`t1.level + 1`). Вычисления прекращаются, когда рекурсивный мост возвращает пустое множество (достигнуты конечные линейные сотрудники).
3. **Формирование финальной витрины верхнего уровня:** Весь откомпилированный массив графа выкатывается наружу. Финальный оператор `ORDER BY level ASC` выстраивает оргструктуру ровными рельсами от высшего руководства вниз к подчиненным.

## ЖУРНАЛ ОТЛАДКИ И ОСОБЕННОСТИ ПРОЕКТИРОВАНИЯ (Debug Log)
В процессе самостоятельной сборки, верификации и тестирования сложного рекурсивного конвейера графов были зафиксированы плановые архитектурные трудности синтаксиса и устранены следующие ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за утери токена объявления):** Падение компилятора базы данных на этапе парсинга из-за пропущенного ключевого слова оператора `AS` и открывающей скобки сразу после объявления рекурсивного имени цеха (`WITH RECURSIVE hierarchy_tree ...`). Магистраль оставалась разомкнутой.
    *   *Метод устранения:* Синтаксический каркас восстановлен в строгом соответствии с промышленным стандартом.
*   **Риск №2 (Ловушка неверной объектной адресации встроенных функций СУБД):** Критическая синтаксическая ошибка на втором этаже рекурсии из-за попытки вызвать скалярный очиститель `TRIM(LOWER())` через точку префикса таблицы (`t2.TRIM`). Встроенные функции СУБД являются глобальными операторами ядра сервера, поэтому префиксы таблиц должны находиться строго внутри аргументов функций.
    *   *Метод устранения:* Адресация скорректирована, физическое поле возвращено на свое место внутри контура: `TRIM(LOWER(t2.emp_name))`.
*   **Риск №3 (Логический перекос рекурсивного моста сопряжения):** Трудности сопоставления внутренних и внешних полей в блоке `ON`, приводившие к бесконечному циклу вычислений и перегрузке RAM сервера.
    *   *Метод устранения:* Контур сопряжения жестко переориентирован на вызов точного выходного алиаса `g1` (`ON t2.manager_id = t1.g1`), что стабилизировало шаг итератора.
*   **Риск №4 (Техническая помарка в финальном ORDER BY):** Механический пропуск символа `R` в команде итогового выравнивания витрины (`ODER BY level ASC`), вызванный высокой скоростью набора текста в Блокноте под административным надзором. Исправлено.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование организационного дашборда)
Очищенная многомерная витрина оргструктуры интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного кадрового контроля.

### Структура дашборда:
1. **Интерактивная древовидная карта холдинга (Organizational Tree View):** Графический иерархический виджет, наглядно отображающий вертикаль подчиненности компании от Генерального директора до линейных специалистов для экспресс-аудита HR-директором.
2. **Индикатор плотности управленческих уровней:** Сводная таблица, отражающая количество сотрудников, зафиксированных на каждом уровне вложенности (`level`), используемая для поиска избыточных промежуточных звеньев.
3. **Панель моментальной фильтрации по руководителям:** Панель поиска, позволяющая моментально изолировать конкретного начальника и отобразить на экране список всех его прямых и косвенных подчиненных.

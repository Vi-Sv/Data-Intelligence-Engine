# Кейс №18: Сквозной аналитический куб «Мосты + Окна» и аудит контрагентов

## Блок: Финтех / Корпоративный аудит
**Заказчик:** Коммерческий директор / Финансовый департамент.

### Бизнес-задача (ТЗ):
> *"Необходимо разработать сквозной аналитический инструмент для оценки финансовой активности контрагентов компании. Требуется консолидировать данные из справочника агентов и лога коммерческих заказов. На итоговую витрину необходимо вывести индивидуальные строки транзакций с указанием имен партнеров и чистой прибыли по каждой операции (пропуски занулить). В смежной третьей колонке требуется рассчитать совокупную историческую прибыль каждого конкретного агента для сопоставления масштаба контрагента с его текущими заказами. Клиенты без транзакций за отчетный период должны быть гарантированно сохранены в отчете с нулевыми показателями. Результаты упорядочить по алфавиту."*

---

## Исходная структура СУБД (Сопряжение один-ко-многим):

### 1. Справочник контрагентов (`agents`):
*   `agent_id` (INT) — Уникальный код партнера в системе холдинга (Первичный ключ).
*   `agent_name` (VARCHAR) — Юридическое наименование контрагента.

### 2. Лог коммерческих заказов (`agent_orders`):
*   `order_id` (INT) — Уникальный номер заказа (Первичный ключ).
*   `agent_id` (INT) — Идентификатор контрагента (Внешний ключ связи).
*   `profit` (INT) — Чистая прибыль от выполнения транзакции.

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе технического анализа грязного ТЗ была применена методология оптимизации «Бритва Оккама» — исключение избыточных вложенных подзапросов и промежуточных группировок для достижения максимальной производительности вычислительного ядра СУБД на Highload-проектах.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Развертывание моста данных (`FROM + LEFT JOIN`):** Движок СУБД берет исходный неприкасаемый справочник `agents` (`t1`) и стыкует с логом оплат `agent_orders` (`t2`) по ключу `agent_ID`. Левостороннее сопряжение гарантирует, что контрагенты без сделок («молчуны») сохраняются в выборке, получая правые поля со значением `NULL`.
2. **Обеспечение качества данных (Data Quality):** Скалярный оператор `COALESCE(t2.profit, 0)` выжигает все внешние пустоты `NULL`, образовавшиеся на стыке пустых связей моста, преобразуя их в красивые числовые нули витрины.
3. **Секционирование аналитического контура:** Конструкция `PARTITION BY t1.agent_name` рассекает сформированную таблицу на герметичные личные зоны расчетов по уникальным текстовым именам контрагентов.
4. **Параллельное оконное суммирование:** Оконный сумматор `SUM` вычисляет общую историческую прибыль внутри каждого выделенного личного сектора и дублирует полученную константу в третью колонку отчета (в Python Pandas данная логика реализована через векторный метод `.groupby().transform('sum')`). Это позволяет обойти деструктивный оператор `GROUP BY`, сохранив 100% детальности лога и атомарную структуру строк.
5. **Финишное выравнивание:** Контур `ORDER BY t1.agent_name ASC` упорядочивает витрину по алфавитному признаку.

## ЖУРНАЛ ОТЛАДКИ И ОПТИМИЗАЦИИ (Debug Log)
В процессе проектирования, верификации и тестирования сквозного конвейера «Мосты + Окна» были зафиксированы и устранены следующие архитектурные риски и проектные решения:

*   **Решение №1 (Оптимизация производительности — Снос подзапросов):** Первоначальный усложненный эскиз архитектуры предполагал расчет макро-сумм через предварительный подзапрос с группировкой и функциями среза дат. Тестирование производительности выявило избыточное потребление ресурсов ОЗУ. Конвейер перепроектирован в один физический проход СУБД посредством связки прямого `LEFT JOIN` и оконного распределения.
*   **Риск №2 (Логический риск потери клиентов-молчунов):** Ошибочное использование внутреннего оператора сопряжения `INNER JOIN` на этапе сборки моста привело бы к автоматическому удалению из аналитического реестра всех зарегистрированных агентов, не совершивших покупок, что критически искажает коммерческий аудит.
    *   *Метод устранения:* Структура жестко зафиксирована на левосторонней отказоустойчивой модели `LEFT JOIN`.
*   **Риск №3 (Защита от падения вычислений):** Наличие скрытых `NULL` в ячейках прибыли лога заказов способно обрушить математику суммирования и выдать пустые значения на витрину.
    *   *Метод устранения:* Внедрены сквозные предохранители `COALESCE`, заменяющие системные пустоты на нейтральный числовой ноль (0) на всех этапах конвейера.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование коммерческого дашборда)
Очищенная сквозная витрина финтех-аудита интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного контроля клиентской активности.

### Структура дашборда:
1. **Матрица сравнительного анализа контрагентов (Pivot Matrix):** Таблица, выводящая связку «Агент -> Прибыль по текущей сделке -> Совокупная историческая прибыль», позволяющая менеджменту за секунду оценивать долю текущего заказа в общем объеме клиента.
2. **Рейтинг лояльности клиентской базы (KPI Card):** Информационный информер, суммирующий общий объем накопленного дохода компании по верифицированным контрагентам.
3. **Фильтр-модуль неактивного сегмента:** Селектор, позволяющий моментально изолировать и выгружать списки «молчунов» (клиентов с нулевой исторической прибылью) для передачи в отдел маркетинга и реактивации.

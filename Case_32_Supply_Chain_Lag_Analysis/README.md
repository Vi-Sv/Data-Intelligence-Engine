# Кейс №32: Оконный ЛАГ-анализ и хронологический шпионаж LAG

## Блок: Промышленная аналитика / Логистика и Снабжение ПТО
**Заказчик:** Начальник отдела логистики / Служба контроля снабжения.

### Бизнес-задача (ТЗ):
> *"Для аудита надежности цепочек поставок, контроля выполнения договорных обязательств контрагентами и выявления системных сбоев ритмичности снабжения необходимо разработать аналитический реестр задержек. Требуется на основе лога поставок сформировать ведомость с выводом уникальных номеров заказов и фактического объема текущего опоздания в днях (пропуски занулить). В смежном третьем столбце необходимо реализовать расчет величины задержки по предыдущему (предыдущему по дате) заказу этого же конкретного поставщика для выявления тенденций к срыву сроков. Наличие пропусков для первых исторических поставок заменить на ноль (0). Данные о детальной структуре строк накладных не должны подвергаться сжатию. Вывод упорядочить по поставщикам и заказам."*

---

## Исходная структура СУБД (Таблица `supply_orders`):
*   `order_id` (INT) — Уникальный номер заказа / транспортной накладной (Первичный ключ).
*   `supplier_id` (INT) — Идентификатор поставщика / контрагента снабжения (Внешний ключ).
*   `delivery_date` (DATE) — Календарная дата фактического прибытия груза на объект.
*   `delay_days` (INT) — Количество дней зафиксированного опоздания относительно графика (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического скрипта была применена модель вертикального ЛАГ-анализа временных рядов через аналитическую функцию смещения назад `LAG()` с секционированием. Данный паттерн позволяет СУБД подсматривать в ретроспективу данных без выполнения тяжелых самообъединений таблицы самой с собой (`Self-Join`), что гарантирует бритвенное быстродействие вычислительного ядра на Highload-нагрузках.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Изоляция среды (`FROM`):** Сервер СУБД открывает физическую таблицу логов логистики `supply_orders` в оперативной памяти (ОЗУ) холдинга.
2. **Обеспечение качества данных (Data Quality):** Функция `COALESCE(delay_days, 0)` перехватывает системные пустоты `NULL` во второй колонке вывода и заменяет их на цифровые нули, подготавливая стерильную базу для верификации.
3. **Развертывание оконных перегородок:** Конструкция `PARTITION BY supplier_id` на лету нарезает массив данных на изолированные герметичные кластеры (виртуальные окна доставщиков) для независимого параллельного счета. Это исключает смешивание транзакционного следа разных компаний.
4. **Фиксация стрелки времени:** Внутри каждого сформированного кластера оператор `ORDER BY delivery_date ASC` выстраивает накладные по строгой хронологической шкале по возрастанию дат. Это гарантирует, что взгляд шпиона направлен строго в прошлое.
5. **Вертикальное извлечение смещения (`LAG`):** Движок СУБД встает на текущую строку, делает шаг назад на 1 позицию по цепи, выкачивает прошлый лог задержки в ОЗУ и приклеивает полученный аргумент в третью колонку отчета (в Python Pandas логика реализована через предварительную сортировку и групповой метод `.shift(1)`). Дефолтный патрон `LAG(..., 1, 0)` автоматически заменяет пустоту отсутствия прошлых записей для первых стартовых заказов на стерильный ноль.
6. **Вывод витрины:** Финальный реестр выравнивается общим оператором `ORDER BY supplier_id, order_id`.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования хронологического конвейера логистики ПТО были зафиксированы и устранены следующие технические риски и зазоры моторики:

*   **Риск №1 (Синтаксическая отсечка из-за капкана кавычек алиаса):** Падение компилятора на этапе парсинга из-за помарки пальцев на входе в объявление имени второй колонки (`AS "Текущее_опаздание),`). Вместо закрывающей двойной кавычки была введена круглая скобка, что разрушило строковый литерал и обрушило синтаксический разбор Postgres.
    *   *Метод устранения:* Структура селекта скорректирована, имя колонки зажато в канонические кавычки `"Текущее_опоздание",` с исправлением орфографии.
*   **Риск №2 (Ловушка избыточных внутренних скобок в функции LAG):** Ошибка компиляции оконной структуры вследствие дублирования скобочного каркаса при упаковке валидатора внутрь аргументов смещения (`LAG((COALESCE(delay_days,0)),1,0)`). Движок выдавал синтаксический отказ.
    *   *Метод устранения:* Контур скобок выровнен в строгий промышленный стандарт: `LAG(COALESCE(delay_days, 0), 1, 0)`.
*   **Риск №3 (Логический риск декомпозиции хронологии — смешивание контрагентов):** Опасность пропуска оператора секционирования `PARTITION BY`, при которой история задержек одного поставщика перетекала бы в начальные строки другого, полностью срывая аудит логистической службы.
    *   *Метод устранения:* Контур окон зацементирован жесткой привязкой к идентификаторам `supplier_id`.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная ЛАГ-витрина снабжения интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для непрерывного контроля ритмичности цепочек.

### Структура дашборда:
1. **Реестр хронологических отклонений логистики (Supplier Delay Tracker):** Детализированная таблица «Заказы -> Текущее опоздание -> Задержка прошлого рейса», подсвечивающая строки с повторными или нарастающими задержками красным цветом для оперативного контроля логистами ПТО.
2. **Диаграмма стабильности поставок (Рейтинг надежности):** Столбчатая диаграмма (Bar Chart), отображающая среднее стандартное отклонение задержек в разрезе каждого поставщика для выявления проблемных контрагентов.
3. **Виджет суммарных простоев объектов:** Цифровой информер, аккумулирующий общее количество дней зафиксированных задержек по критическим материалам.

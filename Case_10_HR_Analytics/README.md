# Кейс №10: Базовое окно агрегации и сепарация филиалов холдинга

## Блок: HR-аналитика / Финтех-Кадры
**Заказчик:** Директор по персоналу.

### Бизнес-задача (ТЗ):
> *"Необходимо сформировать детализированный зарплатный аналитический реестр по всем филиалам холдинга для проведения сравнительного аудита фонда оплаты труда (ФОТ). Требуется вывести персональные идентификаторы сотрудников, нормализованные наименования городов и департаментов, а также скорректированную величину их окладов. В отдельном столбце необходимо рассчитать средний исторический оклад локально по каждому конкретному филиалу (городу) для сопоставления индивидуальных метрик со средними показателями регионального узла. Детализированные строки сотрудников не должны быть агрегированы или скрыты."*

---

## Исходная структура СУБД (Таблица `raw_salaries`):
*   `emp_id` (INT) — Уникальный номер сотрудника (Первичный ключ).
*   `branch_name` (VARCHAR) — Наименование географического филиала (содержит хаотичный регистр ввода и невидимые пробелы).
*   `department` (VARCHAR) — Название структурного подразделения / департамента.
*   `salary` (INT) — Величина оклада сотрудника (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Технический аудит):

В ходе предварительного анализа исходного массива данных `raw_salaries` были локализованы три ключевые уязвимости:
1. **Текстовая неконсистентность географических узлов:** Из-за хаотичного ручного ввода названий городов операторами обычная группировка разорвет единый филиал на независимые некорректные группы;
2. **Наличие системных пустот:** Пропуски `NULL` в ФОТ блокируют корректную работу математических функций сервера;
3. **Конфликт детализации и обобщения:** Необходимость вывести общие средние показатели по городу одновременно с сохранением детальных строк сотрудников без их сжатия.

### Алгоритм построения отказоустойчивого конвейера:
1. **Текстовая нормализация:** При помощи комбинации инструментов `TRIM(LOWER())` (в Python Pandas: `.str.strip().str.lower()`) текстовые наименования городов филиалов и департаментов приводятся к единому регистру, очищаясь от концевых пробелов на входе в конвейер вычислений.
2. **Валидация финансовых метрик:** Оператор `COALESCE(salary, 0)` локализует пропуски данных, принудительно заменяя пустоту нулями, что предохраняет математический движок СУБД от падения.
3. **Развертывание оконного секционирования:** Чтобы приклеить локальную константу среднего оклада филиала к каждой строчке без разрушения детального отчета, применяется оконная функция `AVG` с оператором секционирования `OVER (PARTITION BY)`. Конструкция делит массив данных на изолированные расчетные окна по городам, вычисляет средний оклад внутри каждого филиала и распределяет результат по строкам сотрудников (в Python Pandas данная логика реализована через аналитическую связку `.groupby().transform('mean')`). Результат округляется до сотых долей с помощью `::NUMERIC(10,2)`.
4. **Сортировка витрины:** Финальный отчет выстраивается по алфавитному порядку городов посредством конструкции `ORDER BY ASC`.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования оконного конвейера данных были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за незакрытой скобки):** Падение компилятора базы на строке объявления окна вследствие пропуска парной закрывающей круглой скобки после объявления нормализованного аргумента внутри секции `PARTITION BY`.
    *   *Метод устранения:* Синтаксический каркас восстановлен, парная скобка установлена перед оператором форматирования `::NUMERIC`.
*   **Риск №2 (Косметический заусенец витрины):** Оконный расчет выполнялся корректно, однако из-за пропуска оператора явного приведения типов `::NUMERIC(10,2)` витрина выводила бесконечный неформатированный хвост дробей копеек, снижая читаемость отчета для руководства.
    *   *Метод устранения:* Внедрен инструмент принудительной смены формата для округления средних значений до двух знаков после запятой.
*   **Риск №3 (Логический сбой структуры — тотальное схлопывание данных):** Попытка расчета средних окладов через классический групповой куб `GROUP BY branch_name` на верхнем уровне запроса, что привело к автоматическому уничтожению персональных записей сотрудников и удалению их уникальных идентификаторов `emp_id`.
    *   *Метод устранения:* Конвейер перестроен на отказоустойчивую оконную модель с полной изоляцией вычислений на уровне оконных перегородок `PARTITION BY`.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная зарплатная витрина интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного аудита расходов на персонал.

### Структура дашборда:
1. **Сравнительный маркер ФОТ по регионам (Bar Chart):** Столбчатая диаграмма, отображающая средний оклад филиала в разрезе городов холдинга для быстрого межрегионального анализа рынка труда.
2. **Аналитический реестр отклонений (Data Table):** Интерактивная детальная таблица сотрудников со встроенным вычисляемым полем «Индивидуальный оклад минус средний оклад филиала». Автоматическое условное форматирование подсвечивает сотрудников, чьи оклады выходят за рамки средних нормативов подразделения.
3. **Селектор структурных подразделений:** Фильтр-панель по департаментам для локального среза данных по конкретным цехам и отделам внутри выбранного города.

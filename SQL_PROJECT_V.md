ВВЕДЕНИЕ. ТОПОЛОГИЯ ВСЕЛЕННОЙ ДАННЫХ
СУБД (Система Управления Базами Данных) — это специализированный системный софт (движок), который берет на себя всю грязную физическую работу с диском и RAM. Он преобразует текстовый SQL-запрос в физическое чтение (или запись байтов), распределяет память, блокирует строки от одновременного изменения и следит за сохранностью файлов на сервере.
Весь ИТ-мир работает с данными, но этот мир четко разделен на две разные вселенные. Они преследуют разные цели, и у них разная физика процессов.

ВСЕЛЕННАЯ OLTP (ТРАНЗАКЦИОННЫЕ БАЗЫ / «СБОРЩИКИ»)
Задание – Быстрая, ежесекундная фиксация миллионов мелких операций (купить билет, перевести деньги, обновить статус заказа).
Инструменты – PostgreSQL, MySQL, MS SQL, Oracle.
Логика – Здесь важна строжайшая дисциплина. Таблицы жестко связаны рельсами (PK/FK). База спроектирована так, чтобы ни одна строка не потерялась и не продублировалась. Данные нарезаны на мелкие изолированные острова.

ВСЕЛЕННАЯ OLAP / DWH (АНАЛИТИЧЕСКИЕ ХРАНИЛИЩА / «ВЫЧИСЛИТЕЛИ»)
Цель – Молниеносный расчет тяжелых аналитических отчетов по миллиардам строк за прошлые годы (посчитать средний чек, выявить каскадные задержки, зачистить грязные реестры).
Инструменты – ClickHouse, Greenplum, Snowflake, Snowflake, Postgres-DWH.
Логика – Здесь важна повышенная скорость чтения. Чтобы ее достичь, архитекторы выдергивают физические рельсы (PK/FK), так как проверка связей при загрузке тормозит систему. Данные часто лежат огромными монолитными массивами.

SQL-код универсален для обоих контуров. 
Но: в аналитических DWH-системах готовых рельсов (PK/FK) в схеме нет. Все мосты между таблицами строятся исключительно вручную через неявные связи прямо в теле запроса.
 
Глава 1. ОБЩИЕ ПОЛОЖЕНИЯ
Structured Query Language – Язык структурированных запросов

1. Весь мир SQL – это таблицы с данными, где данные в разных таблицах явно или неявно связаны между собой.
Таблицы — это изолированные острова данных. SQL — это мост между островами. 
Виды связей между таблицами:
Явные связи – прописаны в схеме (как рельсы) – PK/FK (Primary Key / Foreign Key) – ID. (связь может быть «родственная» – одно рождает другое/дружеская – связь независимая»);
Неявные связи – формируются в результате логического сопоставления данных между таблицами, и их последующей комбинированной связке.

2. В таблице, по природе, есть колонка и строка:
Колонка: один столбец = одна характеристика с одним форматом данных (жесткая дисциплина);
Строка: набор характеристик разных форматов, скрепленных в одну горизонталь. В идеальном сценарии имеет ID, но ID — это всего лишь гарант уникальности, нет ID - уникальность формируется комбинацией признаков (комбинацией колонок).

3. Основная роль SQL заключается в приказах, мы приказываем системе показать, что мы хотим видеть, в каком виде (трансформация) предоставить данные, откуда взять эти данные, а также задать фильтрацию – отсев ненужного.
	Иначе говоря: 
SELECT – что мы хотим видеть (вывод данных, математика с данными, смена имен, смена форматов, функции агрегации, оконные функцию), в этом блоке мы создаем виртуальную проекцию (в оперативной памяти RAM), которая не меняет оригинал.
FROM – откуда взять (из какого хранилища);
WHERE – задать фильтрацию (задаем условия, где система режет сырой поток данных, отсеивая не нужное, по заданным условиям до того, как они пройдут в отчет 
Фильтрация в WHERE происходит на этапе чтения сырых строк с диска, строго до того, как данные попадут в блок SELECT на обработку и агрегацию. 


4. Структура построения приказа имеет строгую последовательность:
	БЛОК_1: SELECT (что показать)
	БЛОК_2: FROM (адрес данных)
	БЛОК_3: JOIN ON (виртуальное соединение строк из таблиц: «создание одной сверх-таблицы»)
/JOIN синтаксически не может существовать без FROM. Он является его прямым расширением.
БЛОК_4: WHERE (фильтр данных, или «сито для сырых данных»)
БЛОК_5: GROUP BY (сжатие одноименных строк в одну для проведения расчетов)
БЛОК_6: HAVING (фильтр-весы для групп)
БЛОК_7: ORDER BY (сортировка)
БЛОК_8: LIMIT N (вывод ограниченного числа строк, заданных в N)
/Закон: Пропуск любого блока допустим, но изменение их порядка — запрещено.

5. Система выполняет приказ в определённой последовательности, не равной структуре приказу:
Порядок исполнения запроса системой:
1.	FROM + JOIN ON (Выбор мира данных и его сборка воедино)
2.	WHERE (Первичный отсев до RAM)
3.	GROUP BY (Схлопывание одноименных строк в группы для расчетов)
4.	HAVING (Отсев групп по результатам функций агрегаций) – существует только вместе с GROUP BY)
5.	SELECT (Проекция «выживших» данных», рождение новых имен, математические операции, трансформация, агрегации)
6.	OVER (PARTITION BY….) – (Выполнение оконных функций выполняется после объявления в блоке SELECT, но до ORDER BY) – чтобы фильтровать результат оконных функции, мы выносим их либо в матрешку (SELECT в FROM), либо лаконично – в CTE цех (CTE цех будет далее).
7.	ORDER BY (Упорядочивание)
/Мастер: Задача приказа — сделать так, чтобы до финальной стадии «скульптуры» (SELECT) дошло как можно меньше «камня».

 
ФИЛОСОФИЯ SQL

Визуализируем процесс работы SQL: приказом, мы хирургически срезаем исходную таблицу до того вида, который нужен нам, выполняем некоторые операции трансформации и выводим оставшуюся часть в виде искусственной проекции, не вмешиваясь в исходник («из камня в скульптуру»).
При этом, работа под капотом следующая: приказом мы «запускаем» поток данных, исключаем лишнее нашими действиями, с целью получения точечного результата. Исключение лишнего должно происходить как можно раньше по цепочке выполнения – так, мы стремимся в кратчайшие сроки и не перегружая систему, получить целевой результат.
«SQL — это искусство исключения. Мы не строим отчет, мы исключаем лишние данные, чтобы осталась только истина»
 
Глава 2. БЛОК SELECT 
2.1 ВЫВОД НА ЭКРАН

Инструмент №1: Вывод данных на экран – [SELECT …]

SELECT не просто «показывает» данные, он создает виртуальную проекцию. Он берет отфильтрованный и сгруппированный поток строк и конструирует из них финальные столбцы отчета.
•	Исходная таблица в базе остается нетронутой.
•	Все элементы (колонки, агрегаты, расчеты, окна, подзапросы) перечисляются через запятую (после последнего запятая не ставится).
•	CTE (цеха) объявляются до блока SELECT.
 
Глава 2. БЛОК SELECT
2.2 БАЗОВАЯ МАТЕМАТИКА
- Любая математическая операция обязана быть названа при помощи алиаса AS.

Инструмент №2: Математика в select - базовая – [SELECT … * …]

Математика в SELECT — это вычисление на месте. 
Математика не меняет данные в самом хранилище, она берет вводные данные, выполняет базовые математически операции и выдает результат «виртуально» – в отдельно созданной колонке, математика выполняется исключительно в рамках одной строки (горизонтали):

1. Базовые математические операции (+, -, *, /) между колонок, работают только между столбцами с числовым типом данных. Если типы данных разные — нужен принудительный перевод через CAST.
2. Базовые математические операции (+, -, *, /) между колонками и «вводимым числом» (константой). [пример: SELECT col*100 AS умножение].
3. Базовые математические операции (+, -, *, /) – между ручным вводом/ колонками и результатом агрегаторов. Запрещено напрямую смешивать в одной математической формуле сырую колонку и результат агрегатора. Если в формуле есть агрегат — сырая колонка обязана быть либо в блоке GROUP BY, либо сама обернута в агрегат, либо расчет должен идти через оконную функцию (OVER (PARTITION…).
4. Приоритет и скобки – стандартная математика в контексте приоритетов: сначала умножение и деление, потом сложение и вычитание. Нужно изменить порядок —используем скобки. 
 
Глава 2. БЛОК SELECT 
2.3 МАТЕМАТИКА – ПУСТОТА В СЛОЖЕНИИ И ВЫЧИТАНИИ

Инструмент №3: Предохранитель пустоты (NULL) в сложении и вычитании – маневр с COALESCE.

NULL в SQL – это пустота. Математика с пустотой обнуляет (стирает) весь результат.

[100 + пустота (NULL) = пустота (NULL)]

Чтобы не потерять данные, используем оператор COALESCE (col, 0) 
Этот оператор подставит вместо пустоты – 0, тогда расчет не сломается.

Синтаксис: 
SELECT COALESCE (col, 0)
FROM tab1

//COALESCE превращает все найденные пустоты в указанной колонке (col), в заданное нами значение, в данном примере заменяем пустоту на цифру ноль.
//Расшифровка синтаксиса – оператор COALESCE, «приклеивается» к указанной колонке и заменяет все найденные пустоты, на нужное нам значение, значение строго должно соответствовать формату данных в колонке.
 
Глава 2. БЛОК SELECT 
2.4 МАТЕМАТИКА – НОЛЬ В ДЕЛЕНИИ

Инструмент №4: Защита от деления на ноль – NULLIF.

Без NULLIF один ноль в миллионе строк убьет весь отчет ошибкой «деление на ноль».
NULLIF можно назвать «игнорщиком» - по механизму, он будет игнорировать заданное значение, воспринимая это значение как пустоту.
Но по-научному – NULLIF это оператор сравнения на равенство. 
Его роль в математике — превращать опасное «число ноль» в «безопасную пустоту» и только строго при делении.
Важно помнить: при делении на NULL система не падает, а просто выдает пустоту [NULL]. Мы жертвуем строкой с нулем, чтобы спасти весь миллионный отчет.

Синтаксис:
SELECT 100/NULLIF(col,0)
FROM tab1

Как работает: Мы говорим системе: Сравни значение в указанной колонке с указанным нами значением (в нашем случае «0», однако NULLIF позволяет указывает и другие значения, в том числе не числовые). Если NULLIF находит значение в колонке, которое равно указанному нами значению — он вернет NULL (пустоту). Если они не равны — вернет исходное значение из колонки.

Глава 3. БЛОК ТРАНСФОРМАЦИЯ
3.1 ЗАПЛАТКА ПУСТОТЫ – «КОАЛА»

Инструмент №5: Перехватчик для пустот — [COALESCE]

COALESCE создает дубликат указанной колонки, заменяя все пустоты на новое значение – заплатку.
Иначе, COALESCE — это «заплатка» на дыру. Если система видит пустоту (NULL), она мгновенно меняет её на то, что мы указали в скобках. Принцип: «Если пусто — пиши вот это».

Синтаксис: 
SELECT COALESCE(col, 'замена') AS новая_колонка_с_заменой
FROM tab1

Если в колонке есть данные — система показывает их без изменений.,
Если там NULL — система подставит 'замену'.,
Железный закон типов: Тип значения-заплатки обязан строго соответствовать типу данных самой колонки. В числовую колонку ставим число, в текстовую — текст. Иначе запрос упадет.
 
Глава 3. ТРАНСФОРМАЦИЯ 
3.2 ЗАПЛАТКА ПУСТОТ В СТРОКЕ:
ОПЕРАЦИЯ «КОАЛЫ»

Сценарий существует, если выполняется условие №1 и №2.
№ 1. Если таблица содержит однотипные колонки по смыслу, например:
[факт, проект, норма]
№2. И наша задача заполнить пробелы хоть чем-то.

Синтаксис:
SELECT COALESCE (колонка1, колонка2, колонка3, ‘нет_данных’)
FROM tab1

/Так мы говорим системе проверь колонку 1 на пустоту, если встретил, посмотри в колонку 2, если и там пусто, иди в колонку 3, если и там пусто, ставь мое значение ‘нет_данных’.

COALESCE создает отдельную колонку., выполняя операцию «КОАЛЫ», система движется «слева направо» внутри одной строки (горизонтали) в рамках заданных столбцов, она ищет первую не-пустоту в рамках одного ряда, прежде чем перейти к следующей строке, в конечном итоге формирует одну единственную колонку с данными, полученными в результате выполнения операции.


 
Глава 3. ТРНАСФОРМАЦИЯ 
3.3 МАТЕМАТИКА - МОЩНАЯ АРИФМЕТИКА

Инструмент №6: Округление – [ROUND]
ROUND (col, n) — округление до n знаков после запятой; 
Синтаксис:
SELECT ROUND(col, n)
FROM tab1
Важно зафиксировать, что n может быть не только положительным.
•	ROUND(col, 2) — до сотых (стандарт для денег).
•	ROUND(col, 0) — до целого числа.
•	ROUND(col, -1) — округление до десятков (например, 123 превратится в 120).
/Математический закон: SQL округляет по классическому правилу: 0.5 и выше — в большую сторону, ниже — в меньшую.
Зазор FLOAT: Приблизительные числа (FLOAT) могут округляться некорректно. Для 100% точности в деньгах всегда принудительно приводим колонку к точному типу через CAST(col AS DECIMAL(10,2)) перед округлением.
 
Инструмент №7: убираем минус – [ABS]
ABS (col) — инструмент, который превращает любое отрицательное число в положительное, а положительное оставляет без изменений;
Синтаксис:
SELECT ABS(col)
FROM tab1

 
Инструмент №8: Смена числовых типов данных и усечение разрядов 
[CAST], [::]
Применяется, когда цифра застряла в текстовом формате, когда нужно сменить точность расчетов или жестко отрубить дробный хвост.
Существует два равноправных синтаксиса смены типа («переплавка»):

Классический: CAST(col AS тип_данных) — стандарт СУБД.
Скоростной (Переплавка на ходу): col::тип_данных — Postgres-стандарт, пишется мгновенно на автопилоте рук.
 
ТРИ КОРНЕВЫХ МАТЕМАТИЧЕСКИХ ФОРМАТА
(КАРТА ВСЕЛЕННОЙ ЧИСЕЛ)

I. ЦЕЛЫЕ ЧИСЛА — INT / INTEGER
(INT / INTEGER — полные синонимы. INT сокращение от INTEGER.)
Физика: Числа без дробной части (1, 25, 1000). Занимают минимум места в RAM
Синтаксис на ходу: col::int или col::integer
Жесткий закон округления: Переплавка в INT в PostgreSQL округляет число по правилам математики.
9.9::int – превратится в 10
9.4::int – превратится в 9

II. ТОЧНЫЕ ЧИСЛА (ДЕНЬГИ / СКЛАД) — DECIMAL / NUMERIC
(DECIMAL / NUMERIC Полные близнецы по стандарту SQL ANSI. но numeric в Postgres встречается чаще)
Оба типа созданы для строжайшего коммерческого учета, где округление должно быть предсказуемым, а копейки не имеют права потеряться.
Физика: Числа с фиксированной, гарантированной точностью. 
Синтаксис на ходу: col::decimal(10,2) или col::numeric(10,2)
Параметры: 10 — всего цифр в числе (мощность), 2 — сколько из них выделено под дробный хвост (копейки).

III. ПЛАВАЮЩИЕ ЧИСЛА (ДЛЯ ТЯЖЕЛОЙ АНАЛИТИКИ) — FLOAT / REAL
Синтаксис на ходу: col::float
В чем разница FLOAT / REAL: разница физическая (в емкости «бака» для цифр).
Физика: Это контур приблизительных вычислений. Числа хранятся в двоичном формате, из-за чего компьютер может искажать сотые доли (например, вместо 0.5 физически записать 0.499999).
•	REAL — это легкий бак (4 байта). Хранит до 6–7 значащих цифр.
•	FLOAT (или DOUBLE PRECISION) — это тяжелый бак (8 байт). Хранит до 15 значащих цифр.
Категорически запрещены в фин-отчетах.
 
Инструмент №9 «Топор» — [TRUNC]
Нужен исключительно тогда, когда математическое округление CAST/:: запрещено бизнес-логикой, и нужно жестко обрубить лишний хвост, не прыгая выше головы.

Синтаксис: 
TRUNC(колонка, количество_знаков)

Физика работы: 
TRUNC(9.9, 0) – гарантированно выдаст 9 (топор просто срубил .9).
TRUNC(15.789, 2) – выдаст 15.78 (топор рубит тысячную долю 9, оставив сотые нетронутыми).
 
Глава 3. ТРАНСФОРМАЦИЯ
3.4 СМЕНА ТИПА ДАННЫХ И РАЗВЕДКА

Инструмент №10: Смена типа данных вне математики — [CAST] / [::]
В CAST можно превращать данные в любой тип. Главное правило: данные внутри должны быть совместимы по смыслу (не превратишь слово «Привет» в число, но превратишь строку «100» в INT).
Два синтаксиса: 
Классический CAST(col AS тип) 
Скоростной col::тип (для быстрой переплавки на ходу)
 
ДОСТУПНЫЕ НЕЧИСЛОВЫХ ФОРМАТЫ

VARCHAR / TEXT — Обычные текстовые буквы и символы.
DATE — Чистая дата: только день, месяц и год (2026-06-05). Без часов и минут.
TIMESTAMP — Дата со временем: день, и точное время до секунд (2026-06-05 20:45:00).

ТРИ БОЕВЫХ СЦЕНАРИЯ (РАСКРЫТИЕ РОЛЕЙ)

1.	Текст в Дату / Дата со временем: 
CAST(text_column AS DATE) или CAST(text_column AS TIMESTAMP)
Зачем: Когда при импорте таблицы даты застряли в формате обычного текста.
Роль операции: Переводит буквы в календарный или временной формат СУБД. Без этого система не сможет отфильтровать временной диапазон. Также, переплавляя дату в родной формат, мы можем воспользоваться точечным скальпелем, вырезая только нужное нам значения (день, дату, месяц) – в будущем инструменте EXTRACT (’что_вырезаем’ FROM из_какой_колонки)

2.	Число в Текст: CAST(id_column AS VARCHAR)
Зачем: СУБД физически запрещает склеивать чистые математические числа с текстовыми буквами в функциях сборки текста. (будущий инструмент CONCAT).
Роль операции: Мы принудительно превращаем число (например, ID = 555) в текстовые буквы. Только после этого мы получаем право использовать функцию CONCAT, чтобы собрать красивый номер договора (например: CONCAT('№ ', CAST(id_column AS VARCHAR))).

3.	Дата в Текст: CAST(date_column AS VARCHAR)
Зачем: Когда чистую календарную дату (2026-06-05) нужно разобрать как обычное текстовое слово, чтобы применить к ней строковые фильтры.
Роль операции: Превращает системную дату в плоскую текстовую строку (текст). 
 
Инструмент №11: Разведки качества данных – [CAST]
CAST — это еще и проверка на вшивость. Если при попытке перевести данные одной колонии в какой-либо формат, к примеру CAST(col AS INT) – запрос падает, значит, в данных есть «мусор» (буквы там, где должны быть только цифры).
 
Глава 4. ПОДЗАПРОС 
4.1 ВЛОЖЕННЫЙ (SELECT) В SELECT – НЕ АКТУАЛЬНО
(скалярный подзапрос – инструмент «сверхпозиции»)

Инструмент № X: Сравнение с «мировой константой» – [(SELECT) в SELECT]

Располагая запрос (SELECT …) внутри основного блока SELECT, мы принудительно формируем отдельную колонку, где содержится одно единственное значение для всех строк (курс валют, план на месяц, среднее по больнице).
•	Жесткий закон «Правило 1х1»: Подзапрос внутри SELECT обязан возвращать строго одну колонку и одну строку (ячейку 1х1). Иначе система упадет с критической ошибкой.
•	Защита Мастера: Чтобы «Правило 1х1» не нарушалось, внутри подзапроса всегда используют агрегаторы (AVG, MAX) или LIMIT 1.

КРИТИЧЕСКИЙ СИСТЕМНЫЙ ЗАПРЕТ (ОТК):
Этот инструмент — тяжелый и иррациональный. Если таблица огромная (от 10 000 строк), сервер будет выполнять этот внутренний подзапрос для каждой строки отдельно. Миллион строк = миллион скрытых циклов. Система задохнется.
ПРАВИЛО МАСТЕРА: столкнулся со сценарием «мировой константы» в реальном кейсе? Забудь про этот инструмент. Не пиши подзапрос в SELECT. Выноси этот расчет в изолированный CTE цех (WITH), чтобы вычислить константу один раз и не перегружать процессор СУБД.












Глава 4. ПОДЗАПРОС 
4.2 ВЛОЖЕННЫЙ SELECT В БЛОКЕ WHERE

Инструмент №12: Создание вложенного условия – [(SELECT) в WHERE]

Подзапрос или создание «вложенного условия», существует там, где для вывода данных в отчете, критически важны уточнения, в таком сценарии, таблица главного запроса обращается сама к себе [I], либо к другой таблице [II].

I. В рамках одной таблицы;
/Подзапрос таблицы к себе самой — это легальный способ найти, например, «всех, кто выше среднего» в рамках этой самой таблицы – именно так решаются задачи типа «найди поставщиков, чей объем выше среднего по всей таблице».
II. Когда уточняем у другой таблицы:
Мы «дистанционно» связываемся со второй таблицей, где обязана существовать идентичная колонка из первой таблицы.

Сам формат подзапроса делится на два режима работы:
I. Точечный режим (=, >, <) 
В данном режиме мы жестко ограничены в написании вложенного SELECT, наш запрос внутри должен выводить только одну ячейку (1x1), по которой будет выполняться условия фильтра.
/Мы не можем проводить точечную операцию с массивом, поэтому внутри вложенного select, нельзя пытаться получить массив. (MAX, MIN, AVG, LIMIT – помощники в данном режиме).
II. Режим множества, IN
В данном режиме, жесткие ограничения только в количестве колонок, а именно – одна колонка: (1 x N) полученный результат во вложенном SELECT обязан выводить только одну колонку, но владеть списком строк  – в этом кроется основная суть IN и вложенного SELECT из WHERE – сравнение исходного списка главного запроса и полученного списка, прошедшего условия фильтра в подзапросе, с последующим исключением тех, кто выбыл в подзапросе.
Синтаксис:
SELECT список имён
FROM таблица 1
WHERE КОЛОНКА_ГОНЕЦ IN (SELECT КОЛОНКА_ГОНЕЦ FROM таблица 2.) 
Создавая вложенный запрос через IN, мы буквально срезаем весь список таблицы, оставляя только то, что совпало с тем списком, который был порезан внутри условия вложенного select.

Оба режима могут существовать в обоих сценариях: 
Сценарий 1: В рамках одной таблицы, во вложенном select, в блоке FROM будет исходная таблица., 
Сценарий 2: В рамках общения двух таблиц – во вложенном select, в блоке FROM будет другая таблица., главное условие выполнения сценария IN и двух таблиц: колонка/гонец должна быть и в первой, и во второй таблице.
[IN отправляет список «объектов» на испытание во вложенный запрос, те кто вернулся – стали новой основой для ГЛАВНОГО select запроса]
Ловушка NULL: Если подзапрос вернет список, в котором есть хотя бы один NULL [пустота], весь внешний запрос может вернуть пустоту. Для безопасности в IN подзапросах всегда следует отсекать пустые значения через WHERE col IS NOT NULL.
Этот стоп-кран особенно критичен для оператора NOT IN. В обычном IN (поиск совпадений) одиночный NULL в списке просто проигнорируется системой, но в NOT IN (поиск исключений) один-единственный NULL полностью уничтожит всю выборку. Поэтому страховка IS NOT NULL внутри подзапроса должна быть безусловным рефлексом.
 
Глава 2. БЛОК SELECT: AS (АЛИАС)

11. Инструмент_6: присвоение своего имени – [AS]
Оператор AS – располагается в select, и присваивает имя колонке, математической операции или операции по трансформации – после которой установлен.
Система не учитывает новое название в блоке WHERE, так как он выполняется первее.
Имя в AS пишется без кавычек, но если внутри имени нужен пробел (например, Общая сумма), то все имя обязательно зажимается в двойные кавычки "Общая сумма".
 
Глава 2. БЛОК SELECT: ЯРЛЫКИ (CASE)

12. Инструмент_7: Присвоение имен на базе условий IF-ELSE – [CASE]
CASE располагается в блоке SELECT. CASE создает отдельную колонку (помним про AS), и записывает в соответствующие строки те «значения», которые мы указали в нем и задали условия по присвоению таких значений.
/если в «заданной» колонке найдешь цвет «красный» - присвой имя ‘опасно’, если ‘зеленый’ – ‘безопасно’, иначе – ‘не определено’
Синтаксис:
CASE 
        WHEN колонка_цветов = ‘красный’ THEN 'опасно'
        WHEN колонка_цветов = ‘зеленый’ THEN 'безопасно'
        ELSE 'не определено'
    END AS название_новой_колонки
FROM таблица;
/ Типы данных: Все результаты после THEN и ELSE в рамках одного блока CASE, обязаны быть одного типа (либо все — 'текст', либо все — числа). Нельзя в одной колонке смешивать форматы.
Ключевые детали сборки:
1.	WHEN: Условие к системе;
2.	THEN: Если условие выполняется, система вешает ярлык с соответствующим значением (вводим «что хотим видеть» после then) – система не ограничивает нас в количестве строк THEN.
3.	ELSE: Запасной вариант. Если ни одно WHEN не сработало, запишется это значение. Если ELSE мы не указали — система запишет NULL [пустоту].
4.	END: «Замок», закрывающий логический блок.
5.	AS: Обязательное имя для колонки-результата.
/Внутри WHEN можно использовать и сложные логические связки через AND / OR (например, WHEN цвет = 'красный' AND статус = 'активен' THEN...).
Дополнительный маневр: Вынесение логики классификации в отдельный справочник и подключение через LEFT JOIN (для сотен условий, чтобы не раздувать код).

 
Глава 2. БЛОК SELECT: КОМАНДА ЮВЕЛИРОВ, ВЫСШИЙ УРОВЕНЬ ОЧИСТКИ ДАННЫХ ОТ МУСОРА И КОРРЕКТНОГО ВЫПОЛНЕНИЯ JOIN, WHERE, GROUP BY, А ТАКЖЕ АГРЕГАТОРВ

Все команды ювелиры являются обязательным инструментом мастера при работе с «грязными» таблицами, только благодаря данным командам мы способны очистить таблицу от мусора, который гарантированно влияет на точность вывода данных, проведения расчетов, выполнения условий фильтрации, создания групп и сшивку миров, без них любой запрос неизбежно будет содержать ошибки, с ними и только с ними мы способны буквально спасти миллионы.

13. Инструмент_8: CUT – значит отрезать – [LEFT/RIGHT]
Мы можем создать колонку, в которой от любого значения из указанной колонки, останется n-количество символов от левого или правого конца:
/SELET LEFT(name_col, n), где LEFT – это сторона, с которой будем «отрезать», в скобках указывается колонка, данные которой будем резать, и n – это количество символов, которые система ОСТАВИТ от соответствующего начала.
Как и любая трансформация, CUT (* авторское название) требует AS, иначе заголовок колонки будет техническим и нечитаемым.
Если в ячейке попадется NULL, функции LEFT/RIGHT не упадут в ошибку, они просто вернут NULL.

14. Инструмент_9: Смена регистра – [UPPER/LOWER]
/SELECT UPPER(name_col)
Визуальная трансформация, переводит все данные указанной колонки в ВЕРХНИЙ или нижний регистр. Требует AS.
/Если сделать GROUP BY без смены регистра, система создаст две разные группы для одного и того же материала.
/Чтобы фильтр в WHERE или группировка в GROUP BY сработали идеально, всегда «пропускай» колонку через LOWER или UPPER. Так ты гарантируешь, что ни одна строка не потеряется из-за разного написания.
/При поиске конкретного слова в блоке WHERE приводи к регистру и колонку, и поисковый запрос (например: WHERE LOWER(material) = 'цемент').



15. Инструмент_10: СЦЕПКА – [CONCAT]
CONCAT(name_col1, ‘ ‘, name_col2) – создает отдельную колонку, в которой сцепляет данные из разных столбцов одной горизонтали в одно целое значение, чтобы поставить пробел – его следует заключить в одинарные кавычки.
/ Типы данных: важно помнить, что CONCAT работает с текстом. Если нужно приклеить к тексту число (например, ID заказа), его сначала нужно «расплавить» через CAST(id AS VARCHAR). Современный оператор CONCAT автоматически приводит все числа и даты к тексту
/CONCAT_WS(' ', col1, col2, col3) — продвинутая сцепка с разделителем, где пробел указывается один раз в начале, а не пишется между каждой колонкой.


16. Инструмент_11: Удаление пробелов – [TRIM]
Без TRIM любой идеальный JOIN или WHERE просто промахнутся мимо цели.
TRIM — это автоматический резец, который срезает все паразитные (лишние) пробелы в начале и в конце строки, оставляя только чистое смысловое ядро.
Синтаксис базовый: 
SELECT TRIM(column)
Система сканирует строку с обоих концов. Она видит пробел впереди и пробел позади, а затем «откусывает» их. 
Пробелы между словами внутри строки TRIM никогда не трогает — они защищены.
TRIM работает строго с текстовыми типами данных (VARCHAR, TEXT)
Сценарии применения (Где этот болт держит конструкцию)
Сценарий 1: Броня в блоке WHERE
Сценарий 2: Спасение мостов в блоке JOIN (Комбо-маневр)
/Дешифровка: Ты одновременно принудительно опускаешь регистр букв до маленьких (LOWER(t1.col)) и срезаешь невидимые пробелы с обеих сторон (TRIM(t1.col) в обеих таблицах. Мост становится неуязвим для кривых рук операторов, вводивших данные.
Синтаксис специализированный (встречается реже):
•	LTRIM(column) — режет пробелы только слева 
•	RTRIM(column) — режет пробелы только справа


 
Глава 3. БЛОК FROM: ТОЧКА ВХОДА И МАТЕРИАЛИЗАЦИЯ ПРОСТРАНСТВА

17. Инструмент_12: Адрес хранилища данных – [FROM]
Логика: FROM — это приказ системе определить физический адрес данных. 
Таблицы в базе — это изолированные острова, и через FROM мы указываем, на какой конкретно остров высаживается «поисковый отряд». Без этого блока ни одно вычисление в SELECT или отсев в WHERE невозможны — системе просто не к чему будет приложить логику.
Синтаксис (Базовый): 
SELECT col FROM table
Синтаксис (AS архитектурный): 
FROM very_long_name AS t1
Закон Первородства: несмотря на то, что в коде, FROM мы пишем на второй строке (после SELECT), система под капотом выполняет его первым. Это абсолютное начало координат запроса, где присвоенный алиас автоматически становится «законом» для всех остальных блоков: и для SELECT, и для WHERE, и для GROUP BY.
1.	Архитектурный Алиас (AS): Присвоение короткого «позывного» таблице прямо в блоке FROM.
o	Как только мы прописали FROM orders AS o, старое длинное имя orders перестает существовать для системы в рамках этого запроса.
2.	Точечная адресация (Префиксы): Алиас во FROM обязывает обращаться к колонкам через точку: o.status, o.price. Это исключает «амбивалентность» (двусмысленность) для сервера, когда в запросе появится больше одной таблицы.




 
Глава 3. БЛОК FROM, ПРОИЗВОДНАЯ ТАБЛИЦА

18. Инструмент_13: Чистая таблица, [FROM (SELECT) AS]
FROM (SELECT) AS – «Создание временной, предварительно обработанной таблицы прямо в воздухе».
По умолчанию FROM берет «грязную» таблицу из базы, где не всегда, для анализа, нужны и 50% всего массива данных.
Чтобы «отрезать гнилой кусок пирога», до начала хирургической операции по извлечению данных, система позволяет из обычного запроса SELECT, сделать чистую основу для ГЛАВНОГО SELECT запроса. 
Для этого, диктуем обычный запрос: 
SELECT city, AVG(salary) AS average salary 
FROM workers
GROUP BY city
Прячем его в скобки, подкладываем во FROM ГЛАВНОГО SELECT запроса, и обязательно присваиваем имя, иначе сервер намертво заблокирует запрос с ошибкой (таблицы не существует) – помним, мы создаем новую таблицу при помощи запроса, помещенного в FROM, такой таблицы в базе не существует, поэтому ее мы обязаны назвать.
Полный синтаксис:
SELECT t1.city, t1.average_salary
FROM (
    SELECT city, AVG(salary) AS average_salary
    FROM workers
    GROUP BY city
) AS t1
WHERE t1.average_salary > 100000;
/Для внешнего (главного) запроса, подзапрос из FROM выглядит как самая обычная чистая таблица.
Сценарии применения (Когда без этого не обойтись):
Двухуровневая агрегация – проще говоря, нужен тогда, когда с первым результатом какой-либо агрегации нужно выполнить еще одну функцию агрегации.
/Будущая альтернатива – CTE цеха которые делают код чище и читаемые, избавляя от громоздких вложенных скобок во FROM (их мы разберем дальше).

 
Глава 3. БЛОК FROM, МОСТ МЕЖДУ МАТРЕШКАМИ

19. Инструмент_14: Соединение двух чистых таблиц и их связка
Соединение двух независимых подзапросов из FROM в один отчет.
Архитектура Highload-запросов (высоконагруженных систем). Это стыковка двух виртуальных вселенных, созданных на лету.
В аналитике это применяется постоянно, когда нужно объединить данные из двух тяжелых таблиц, но ты не хочешь тащить в мост (JOIN) миллионы сырых строк. Ты сначала «сушишь» каждую таблицу отдельно в свой «чистый пирог» во FROM, а потом соединяешь эти два чистых результата.
Архитектурный чертеж (Синтаксис стыковки двух подзапросов)
 
Точка опоры (ON): Колонки, по которым идет сцепка, должны быть явно выведены внутри обоих подзапросов (если ты забудешь написать city во внутреннем SELECT хотя бы одного подзапроса, внешний мост не построится).
Этот маневр экономит прорву оперативной памяти сервера, потому что ты соединяешь не миллионы сырых строк, а, например, всего 10 строк городов с 10 строками городов
 
Глава 3. БЛОК FROM, ПРИСВОЕНИЕ ИМЕН

20. Инструмент_15: Системный позывной таблицы — [AS в блоке FROM, JOIN]
Логика: Когда мы вызываем таблицу из хранилища, мы вешаем на неё короткий ярлык (алиас). Это нужно для того, чтобы во всех остальных частях приказа (SELECT, WHERE, JOIN) не писать длинное «имя» таблицы, а обращаться к ней по короткому позывному.
Синтаксис: FROM very_long_table_name AS t1
1.	Закон Приоритета: AS в блоке FROM — это «первичный кирпич». Он рождается в самом начале исполнения запроса, поэтому его видят абсолютно все блоки, стоящие выше или ниже.
2.	Диктатура имени: как только мы присвоили таблице имя (например, t1), старое длинное имя для системы перестает существовать. 

JOIN и AS.
Когда таблиц становится больше одной, синтаксис превращается в цепочку: каждая новая таблица заходит через приказ JOIN и сразу получает свой позывной. Синтаксис присвоения имен нескольким таблицам:
FROM table_leader AS t1
JOIN table_partner AS t2 ON t1.id = t2.parent_id
JOIN table_extra AS t3 ON t2.id = t3.partner_id
Разбор:
1.	Первый позывной: Задается в блоке FROM. Это «база», к которой мы всё приклеиваем.
2.	Последующие позывные: Задаются сразу после названия таблицы в каждом блоке JOIN.









Глава 4. БЛОК JOIN: СТЫКОВКА МИРОВ

Логика: Этот блок создает временный мост между таблицей-лидером (в блоке FROM) и таблицей-партнером. Стыковка происходит строго горизонтально: система берет строку из левого мира, ищет совпадение в правом мире и склеивает их в одну длинную супер-строку. По сути, мы можем входить в FROM с любой таблицы, связка произойдет, но таблица в FROM приобретает статус лидера, что важно для JOIN LEFT.

21. Инструмент_16: Мост пар – [INNER JOIN ON]
Этот мост пропускает в итоговый отчет только тех, у кого есть стопроцентная взаимность (пара). Если строка из левой таблицы не нашла себе пару в правой (или наоборот) — она уничтожается и полностью исчезает из финального отчета.
Синтаксис:
SELECT o.id, o.price, c.client_name
FROM orders AS o
INNER JOIN clients AS c ON o.client_id = c.id
INNER – «Оставь только те строки, которые нашли свою пару»
ON – Точка опоры: приказ, который задает условие сцепки. Если значение из таблицы-лидера, нашло свою копию в таблице-партнёре, связь между строками двух таблицу существует, и формирует сверх-строку в рамках сформированной сверх-таблицы.
Сверх-таблица –это объединение таблиц в одну большую.
/Ключевое слово INNER можно опускать (просто написать JOIN), СУБД по умолчанию воспримет это как INNER JOIN.
 

Глава 4. БЛОК JOIN: НЕУБЕВАЕМАЯ ТАБЛИЦА ЛИДЕР

22. Инструмент_17: LEFT JOIN — Левый мост (Удержание контроля).
Логика: Самый востребованный инструмент аналитика. Этот мост защищает левую таблицу (лидера) от любых потерь. Все строки из таблицы во FROM гарантированно выживут и попадут в отчет. Если строка из правого мира не нашла пару — мост все равно строится, но на месте правых данных система оставляет безопасную пустоту (NULL).
Синтаксис:
SELECT c.client_name, o.id AS order_id
FROM clients AS c
LEFT JOIN orders AS o ON c.id = o.client_id;
Механика NULL: Если клиент «Иванов» ничего не купил, он все равно останется в отчете. Просто в колонке order_id напротив него будет написано NULL.
Смысл: Идеально для поиска просадок и долгов. Например: взять список всех плановых задач (LEFT JOIN факт выполнения) -> где факт IS NULL -> мгновенно получили список сорванных этапов.
Капкан №1: Раздувание строк (Брешь-контроль)
Брешь-контроль (Главный капкан): JOIN осуществляет сцепку «строка к строке». Если в правой таблице по точке опоры (ON) найдется не одна строка, а несколько, система продублирует левую строку под каждую правую. Отчет раздуется, а все будущие расчеты из агрегаторов увеличатся в разы. Поэтому точка опоры в правой таблице *в большинстве случаев (90%) должна быть уникальным ключом (PK ID).
Защита: Точка опоры справа в 90% случаев должна быть уникальным ID (Primary Key).
Исключение (Сценарий «Один-ко-многим»): Размножение строк — это норма, если нужна детализация (например: один Договор -> LEFT JOIN пять Этапов оплат). Главное — помнить о размножении при дальнейших расчетах.
 
Капкан №2: Превращение в INNER (Стоп-кран)
Если после LEFT JOIN ты фильтруешь правую таблицу в общем блоке WHERE, мягкий мост ломается и превращается в жесткий INNER JOIN. Система выкидывает строки с NULL, уничтожая левого лидера.
Три щита защиты (Как не сломать мост):
1.	Маневр через NULL в WHERE (Самый простой):
WHERE o.status = 'Оплачено' OR o.status IS NULL
2.	Маневр через COALESCE в WHERE (Зануление пустот)
WHERE COALESCE(o.amount, 0) > 100000; -- NULL превратится в 0 и строка выживет
3.	Маневр через AND в блоке ON (Профессиональный):
LEFT JOIN orders AS o ON c.id = o.client_id AND o.status = 'Оплачено';

Срез-шпаргалка:
•	INNER JOIN: Оставляет только пересечение. Нет пары — строка уничтожается.
•	LEFT JOIN: Сохраняет левую таблицу целиком. Нет пары справа — подставляет NULL (пустоту).
•	ON: Условие сцепки. Требует точного совпадения типов данных.
•	Префиксы (t1.col): Жесткий закон для исключения конфликта имен колонок.

 
Глава 4. БЛОК JOIN: НЕЯВНЫЕ СВЯЗИ

23. Инструмент_18: Создание уникального ID вручную – [JOIN ON (AND)]
В IT этот маневр называется составным соединением (Composite Join) или неявным маппингом. Твоя задача здесь — найти несколько общих точек опоры в разных таблицах, спаять их вместе и заставить систему поверить, что эта комбинация и есть уникальный ID.
Глубинный смысл: Стыковка по смысловым координатам
Представь две таблицы из ПТО. Первая — «Снабжение» (где ведут учет прихода материалов), вторая — «Сметы» (где заложены лимиты). Программисты не заложили в базу сквозной material_id. Всё, что у тебя есть — это текстовые названия и характеристики.
Если ты попробуешь сделать JOIN только по названию материала (ON t1.name = t2.name), твой мост рухнет в хаос дубликатов, потому что слово «Цемент» или «Арматура» встречается в таблицах сотни раз для разных марок и объектов.
Решение: Мы строим мост не по одной опоре (ID), а по целой системе опорных координат. Мы говорим системе: «Строка левого мира равна строке правого мира только тогда, когда совпадает – И материал, И его марка, И объект строительства».
/Создание уникального ID кустарным методом
Архитектурный чертеж (Синтаксис Composite JOIN) 
SELECT 
    s.object_name,
    s.material_name,
    s.material_grade,
    s.quantity_fact,
    e.quantity_estimate
FROM supply AS s
LEFT JOIN estimates AS e 
    ON  s.object_name   = e.object_name 
    AND s.material_name = e.material_name 
    AND s.material_grade = e.material_grade;
Оператор AND в блоке ON – Это главный замок. Он заставляет систему проверять все условия одновременно. Строка справа приклеится только если совпадут все три параметра.
Общая альтернатива: Сборка суррогатных текстовых ключей-хэшей через CONCAT внутри CTE перед соединением.
 
Глава 4. БЛОК JOIN: НЕЯВНЫЕ СВЯЗИ – КАПКАНЫ

Главные капканы сценария JOIN без ID 
Капкан 1: Невидимые пробелы и разный регистр
Текст — коварная материя. Если в первой таблице написано 'ЖК Центр', а во второй 'ЖК Центр ' (с невидимым пробелом на конце) или 'жк центр' (с маленькой буквы), система скажет: «Не совпало» — и LEFT JOIN выкинет NULL.
•	Защита (Инструменты ювелира в блоке ON): Мы принудительно зачищаем текст прямо в точке сцепки с помощью LOWER и функций удаления пробелов (TRIM):
ON TRIM(LOWER(s.material_name)) = TRIM(LOWER(e.material_name)) – эталонный стандарт чистого маппинга.
Капкан 2: Размножение данных (Дубликаты)
Если даже при трех условиях в правой таблице найдутся две абсолютно одинаковые строки (например, две сметные строки на Цемент М500 для одного объекта), система размножит твою левую строку.
Синтаксическая дотошность:
В блоке ON можно использовать сколько угодно операторов AND, связывая текстовые, числовые колонки или даты. Главное — типы данных в каждой паре должны совпадать (текст сравниваем с текстом, число с числом).
Капкан 3: Пустоты
Чтобы составной мост не разбился о пустоту, текстовые NULL нужно принудительно перехватывать через «Коалу» прямо внутри условий ON:
AND COALESCE(s.material_grade, 'нет') = COALESCE(e.material_grade, 'нет')
Логика: Если с обеих сторон прилетят пустоты, COALESCE превратит их в одинаковое слово 'нет'. Условие 'нет' = 'нет' выдаст TRUE, замок закроется, и мост выстоит.

 
Глава 5. БЛОК WHERE: ФИЛЬТРАЦИЯ

24. Инструмент_19: Фильтрация [WHERE]. 
Самый ключевой и приоритетный блок.
WHERE «условия»
Через данный блок мы просеиваем данные, задавая им условия по выводу истины и исключения лишнего.
Помни: WHERE выполняется до SELECT, нельзя фильтровать в WHERE по новым именам колонок, которые создали в SELECT.
Инструментарий: В WHERE используются мы можем использовать простейшие математические операции: (=, >, <, <=, >=), а также мощные операторы: IN (список), BETWEEN (диапазон), LIKE (поиск по маске) и IS NULL (проверка на пустоту).


 
Глава 5. БЛОК WHERE: СОЧЕТАНИЕ УСЛОВИЙ

25. Инструмент_20: Перечисление условий, [AND]
[AND – сужает сито]
В блоке WHERE запятая синтаксически запрещена. Если нужно, чтобы запись прошла через фильтр по нескольким критериям одновременно, единственный путь — логическая связка условий при помощи оператора AND.
Синтаксис: 
WHERE условие_1 AND условие_2
Глубинный смысл: AND — это умножение вероятностей. Чтобы строка выжила, она должна соответствовать всем истинам одновременно.

26. Инструмент_21: Логическое ИЛИ в блоке where, [OR]
[OR – расширяет сито]
Если применяется только OR, диктуем системе «или выполняется условие_1 или выполняется условие 2»;
Если выполняется совместно с AND, расширяет выборку (в отличие от чистого AND, который её сужает).
Приоритет: В сложных запросах AND выполняется раньше, чем OR. Чтобы задать свой порядок, используем скобки.
Синтаксис: 
WHERE (условие_1 OR условие_2) AND условие_3.
Если много OR по одной и той же колонке – лучше применить IN (1, 2, 3). Это есть «скрытый OR», который пишется быстрее.
Маневр профи (Ловушка NULL): если нужно видеть пустые записи в отчете. Всегда добавляй OR колонка IS NULL;









Глава 5. БЛОК WHERE: ИНСТРУМЕНТАРИЙ В WHERE

27. Инструмент_22: Поиск по корню слова, [LIKE]
Оператор [LIKE] – это оператор «совпадения», применяется если нужно осуществить – «поиск по фрагменту/части слова», иначе можно сказать «поиск по КОРНЮ слова с разными приставками и окончаниями. 
Синтаксис: WHERE КОЛОНКА_ДЛЯ_ПОИСКА LIKE ‘%КОРЕНЬ%‘
Пример:
WHERE ИСКОМЫЙ LIKE ‘%РОД%‘
В результате, система покажет из колонки ИСКОМЫЙ только то, что включает в себя «корень» РОД, например: РОДительсикй, РОДной, РОДина, предРОДовой и т.д.
Чтобы система игнорировала ПРИСТАВКУ, % (символ-джокер) ставится перед словом; 
Чтобы система игнорировала ОКОНЧАНИЕ, % ставится после корня.
Маневр профи (Регистр): Оператор LIKE чувствителен к регистру, 'Род' и 'род' — для него разные миры.
Решение:  используем ILIKE – он игнорирует высоту букв и находит 'РОД', 'род' и 'Род' одинаково успешно.
 
28. Инструмент_23: КПП – [оператор IN]
Оператор IN («ВКЛЮЧИТЬ В СПИСОК») — это «быстрый фильтр» для колонки, в которой нам нужно отобрать несколько конкретных значений. Вместо того чтобы писать много OR, мы создаем контрольный список.
Синтаксис:
WHERE колонка_с_данными IN ('значение_1', 'значение_2', 'значение_3')
Система пропустит только те строки (всю горизонталь), чье указанное имя совпало с контрольным списком указанных имен.

Если нужно исключить конкретный список – применяем NOT IN («ИСКЛЮЧИТЬ ИЗ СПИСКА»). Например, «Покажи все диаметры, кроме 57 и 89». Это моментально очищает отчет от ненужного шума.
Важно - одна пустота в NOT IN обнулит весь отчет. 
Как занулить этот капкан: принудительная очистка списка от NULL прямо в условиях – добавь жесткое условие IS NOT NULL перед списком
WHERE diameter IS NOT NULL AND diameter NOT IN (57, 89) 
Логика движка: Мы сначала убираем из проверки все пустоты (IS NOT NULL), а затем спокойно прогоняем чистые числа через КПП. Запрос спасен.
 
29. Инструмент_24: Оператор фильтра «между» — [BETWEEN]
Оставь только значения ОТ и ДО. Корректно работает с числами и датами, с текстом будут проблемы.
Главная особенность: Границы включены в результат. Если ищешь BETWEEN 10 AND 20, то и 10, и 20 попадут в отчет.
Синтаксис:
WHERE колонка_с_данными BETWEEN значение_1 AND значение_2
Применение в разных мирах:
1. Числа: WHERE price BETWEEN 100 AND 500 (всё, что стоит от 100 до 500 включительно).
2. Даты: WHERE date BETWEEN '2023-01-01' AND '2023-01-31' (весь январь).
Если нужно найти внешние значения, используют NOT BETWEEN. 
Например: WHERE price NOT BETWEEN 10 AND 1000 — это мгновенный поиск слишком дешевых или неоправданно дорогих товаров.
 
Глава 5. БЛОК WHERE: IS NULL – ОХОТНИК ЗА ПУСТОТОЙ

В SQL NULL — это не ноль и не пробел, это «неизвестность» – пустота. Поэтому обычные знаки сравнения (=, <, >) с ним не работают. Чтобы поймать пустоту, нужен специальный «детектор».

30. Инструмент_25: Охотник за пустотой — [IS NULL]
IS NULL – это оператор для поиска строк, где данные не были введены (пустота).

Критическое правило: Никогда не пишут WHERE column = NULL. Это логическая ошибка, которая вернет «ничего». Правильно только IS NULL.

Два режима охоты IS NULL:
1. IS NULL (Поиск дыр) – Находит строки, где значение отсутствует.
Пример: WHERE email IS NULL (найти пользователей, которые не оставили почту).

2. IS NOT NULL (Поиск наполненности) – Находит только заполненные строки.
Пример: WHERE phone IS NOT NULL (выгрузить только тех, кому можно позвонить).

Маневр профи: «Сверка качества»: 
Используем IS NULL вместе с COUNT(*), чтобы понять масштаб пустот в данных.
SELECT COUNT(*) FROM table WHERE column IS NULL — это быстрый замер «пустот» в конкретной колонке.
Глубинный смысл: NULL в SQL — это черная дыра. Все, что соприкасается с ней в математике, исчезает (10 + NULL = NULL). IS NULL — это единственный «фонарик», который позволяет увидеть эти дыры до того, как они испортят расчеты.

Обо что разбивается IS NULL при работе с текстом — если в ячейках сидят «голые» кавычки ('') или пробел ' ', то для сервера эта ячейка НЕ является NULL. Она заполнена символом пустоты. В итоге WHERE col IS NULL пролетит мимо этой строки и не поймает её. Чтобы гарантированно запереть этот капкан, всегда стоит применять страховку: 
WHERE (col IS NULL OR TRIM(col) = '').


 
Глава 6. БЛОК GROUP BY

31. Инструмент_26: Cоздание групп – [GROUP BY и одна колонка]
GROUP BY col
При помощи GROUP BY, несколько одноименных строк, из указанной колонки, сводятся в одну. Истинно работает в связке с функциями агрегации.
Жёсткое правило: если в SELECT есть хотя бы одна агрегатная функция (например, SUM), то все остальные колонки в SELECT обязаны быть перечислены в GROUP BY.
Вместо длинного имени колонки в GROUP BY можно писать её номер из SELECT. Пример: 
SELECT city, SUM (sales) ... GROUP BY 1.

NULL в группах: Помни, что GROUP BY считает все NULL одинаковыми и соберет их в одну отдельную группу «Неизвестно»]

32. Инструмент_27: Создание групп из групп (группы внутри группы) – [группы в группе]
Выполняется, если среди одной группы имен, нужно выделить микро – группы по какому-либо признаку (ВТОРОЕ ИМЯ или более 2х имен);
Система комбинирует имена групп и уникальные ВТОРЫЕ ИМЕНА (признаки) внутри группы, выделяя каждой комбинации свою отдельную строку.
[GROUP BY city, month. Тогда система создаст уникальные пары (Москва-Январь, Москва-Февраль). Это позволяет детализировать отчет, не теряя мощности агрегации.]

Синтаксис: 
…GROUP BY имя_колонки_1, имя_колонки_2… 
Все колонки, указанные в GROUP BY обязаны дублироваться в SELECT.
Если ты группируешь по Городу и Месяцу, но в SELECT вывел только Город — ты увидишь несколько строк «Москва» с разными суммами и не поймешь, почему они не схлопнулись (а они разбиты по месяцам, которых не видно).
Порядок колонок в GROUP BY важен для структуры «вложенности» (сначала большая группа, потом подгруппа).
 
Глава 7. КОМБИНАЦИЯ SELECT, GROUP BY АГРЕГАТОРЫ

33. Функции агрегации (агрегаторы) – располагаются в select, каждая функция агрегации «создает» отдельно существующую колонку со своим результатом, которая обязаны быть названа, применив AS.
Если в SELECT есть хотя бы одна агрегатная функция (например, SUM), то все остальные колонки в SELECT обязаны быть перечислены в GROUP BY.

Важные нюансы:
– Агрегатные функции игнорируют NULL (все кроме COUNT (col));
– Нельзя выводить обычную колонку рядом с агрегатом без GROUP BY;

Глава 8. АГРЕГАТОРЫ

34. Инструмент_28: Посчитать количество строк/ID – [COUNT (*)]
[посчитать количество строк]
COUNT (*) – считает ВСЕ строки, считая и пустые NULL – таким образом считает факт «жизни» строки;
– Указывая (*), система не привязывается к колонке;
– Выполняется в последнюю очередь, а значит, что считает все то, что осталось в выводимой таблице;
Истинно работает в связке с GROUP BY: 
– Без GROUP BY – считает вообще все строки;
– С GROUP BY считает количество строк для каждой созданной группы.

Маневр профи: Сравнивай COUNT(*) и COUNT(col). Если цифры разные — ты нашел «дыры» (NULL) в данных без ручного перебора.
 
35. Инструмент_29: Посчитать количество записей в указанной колонке
COUNT (имя_колонки) — это детектор наличия факта. Если в ячейке пустота (NULL), для этой функции её не существует. 
[пропускает пустые (NULL), а значит делает акцент на записи в строке, а не на факте существования самой строки.]
Наглядный пример:
SELECT COUNT(email) AS registered_users - Посчитает только тех, у кого введен email
FROM users;

36. Инструмент_30: Узнать количество уникальных строк – [COUNT (DISTINCT)]
Применяется, если нужно узнать количество только уникальных «имен» в указанной колонке, 
Можно сказать: «игнорирует повторения», 
Если применили без GROUP BY - считает количество потенциальных групп в заявленной колонке.
Если применили с GROUP BY – считает уникальных, в группах.
COUNT (DISTINCT column) игнорирует NULL.

37. Инструмент_31: Собрать сумму – [SUM (col)]
Применяя SUM, мы обязаны указать колонку;
SUM работает только с числовым типом данных.
Если применили без group by – просуммирует все то, что располагается в указанной колонке, и выдаст результат;
Если применили с group by – просуммирует данные в созданных группах, у каждой группы свой собственный результат, данная функция сотрудничает с каждой группой отдельно, не перемешивая данные между собой., 
Внутри SUM можно проводить математику до суммирования.
Пример: SUM (price * quantity).
 
38. Инструмент_32:Найти среднее – [AVG]
SELECT … AVG(col)
AVG (col) – ищет среднее значение из указанной колонки, складывает все значения и делит на общее количество, при этом игнорирует пустоты (NULL), поэтому, чтобы результат нахождения среднего был истинным, применяем COALESCE, где пустышки просто заменяем на ноль – это выровняет среднее, охватив вообще все.
Синтаксис:
SELECT имя_колонки, AVG (COALESCE(имя_колонка, 0)) AS СРЕДНЕЕ_ЗНАЧЕНИЕ
FROM table
GROUP BY имя_колонки
Без group by просто выдаст среднее на всю таблицу из заявленной колонки;
С group by выдаст среднее по каждой группе, аналогично с группами в группе., сотрудничает с каждой группой отдельно.

Маневр профи:
Сравнивание AVG(col) и AVG(COALESCE(col, 0)). Если разница велика, значит, в системе много «пустышек», которые сильно завышают (или занижают) реальную картину эффективности.

39. Инструмент_33: Найти самое максимальное/минимальное [MAX/MIN]
MAX/MIN возвращает одно единственное значение из указанной колонке, либо самое MAX, либо самое MIN.
SELECT колонка, MAX (колонка) AS самое_большое
FROM table
GROUP BY колонка
Без group by, покажет самое высокое значение из заявленной колонки,
С group by, покажет самое высокое значение в каждой группе., 

Типы данных: В отличие от SUM,  – MAX и MIN работают с датами и текстом.
MAX для дат — это самая свежая дата;
MAX для текста — последнее слово по алфавиту;

Если нужна вся строка, в которой находится максимальное значение (например, имя самого дорогого товара), обычный MAX не поможет. Придется использовать ORDER BY цена DESC LIMIT 1

Глава 9. БЛОК ORDER BY: СОРТИРОВКА

40. Инструмент_34: Провести сортировку, [ORDER BY]
	Сортировка выполняется в самом конце, когда все процессы в запросе выполнились, и нужно просто причесать результат., сортировку можно выполнить и в фоне, по другой колонке, тогда тот порядок, в котором выстроятся строки/ID в фоне, будет проецироваться на выводимый результат.
Синтаксис:
SELECT колонка_с_данными
FROM ТАБЛИЦА
ORDER BY колонка_с_данными DESC/ASC

Срез угла в ORDER BY: Можно ставить порядковый номер колонки из SELECT. Например: ORDER BY 1 DESC (сортировка по первому столбцу).
ORDER BY идет после SELECT, можно сортировать по новым именам и вычислениям из SELECT.

NULL в сортировке: По умолчанию NULL часто считаются самыми маленькими (в начале при ASC) или самыми большими (в начале при DESC). Если они мешают, используют надстройку NULLS LAST (в конец) или NULLS FIRST (в начало).
Синтаксис простой: надстройка ставится в самом конце блока ORDER BY, сразу после направления сортировки (ASC или DESC).
ORDER BY колонка [ASC/DESC] NULLS LAST -- Все пустые строки упадут в самый низ
ORDER BY колонка [ASC/DESC] NULLS FIRST -- Все пустые строки всплывут в самый верх

Можно сортировать по логическому условию. Например: ORDER BY (цена > 1000) DESC. Это выкинет все дорогие товары в начало списка, а внутри них уже пойдет остальная сортировка.

Множественная сортировка: Можно сортировать по нескольким колонкам сразу: ORDER BY 1 DESC, 2 ASC. Сначала сработает первый приоритет, затем внутри него — второй. Важно – для каждого из множества сортировки требуется уточнение напраления.
 
Глава 10. БЛОК HAVING: ВЕСЫ ДЛЯ ГРУПП

41. Инструмент_35: Весовое сито групп – [HAVING]
HAVING – это фильтр, который работает строго со сгруппированными данными после GROUP BY.
Синтаксис:
SELECT city, SUM(sales) AS total_sales
FROM orders
GROUP BY city
HAVING SUM(sales) > 100000;

Глубинный смысл:
WHERE режет сырые строки до группировки. HAVING взвешивает уже готовые группы по результатам их агрегатов (SUM, COUNT, AVG).
Жесткое правило:
В HAVING нельзя использовать обычные колонки без агрегатора. Все обычные признаки (status, object_id) обязаны фильтроваться раньше — в блоке WHERE.
Маневр профи:
Внутри HAVING нужно писать агрегатную функцию целиком 
HAVING SUM(sales) > 10
Алиас из SELECT система там не увидит, так как HAVING выполняется раньше SELECT.
 
Глава 11. БЛОК SELECT: UNION
ВЕРТИКАЛЬНЫЙ МОСТ: ОБЪЕДИНЕНИЕ СТРОК

42. Инструмент_36: Таблица над таблицей — [UNION / UNION ALL]
Оператор UNION располагается между двумя независимыми приказами SELECT и склеивает их результаты вертикально (добавляет строки вниз, увеличивая длину отчета).
Синтаксис:
SELECT material, quantity 
FROM supply_2024

UNION ALL

SELECT material, quantity 
FROM supply_2025

Существует два режима сшивки:
1. UNION ALL — Просто приклеивает нижние строки к верхним. Все дубликаты выживают. Работает мгновенно.
2. UNION — Склеивает строки и принудительно удаляет полные дубликаты. Работает медленно (нагружает процессор).

Жесткие законы сборки:
Закон одинакового количества: Число колонок в верхнем и нижнем SELECT должно быть строго равным.
Закон однородных типов: Столбцы на одинаковых позициях должны совпадать по формату (число под числом, текст под текстом).
Диктатура имени: Заголовки колонок в итоговом отчете берутся строго из первого (верхнего) SELECT.
Если нужно упорядочить весь итоговый склеенный отчет (например, по дате или цене), блок ORDER BY пишется всего один раз в самом конце.

 
Глава 12. БЛОК SELECT: ТОЧКА СИНХРОНИЗАЦИИ NOW

43. Инструмент_37: Точка синхронизации, [NOW () в SELECT]
Функция NOW(), расположенная в SELECT, создает колонку и выводит одно единственное значение – дату и время актуального запроса. Нужно, если отчет нужно зафиксировать как контрольную точку проверки. 
Либо для сравнения значений из прошлых дат (взгляд назад)
Либо для сравнения из будущего с настоящей контрольной точкой синхронизации (взгляд из будущего).
Функция NOW() располагается в блоке SELECT или WHERE и мгновенно возвращает текущую дату и точное время сервера на момент запуска запроса в формате YYYY-MM-DD HH:MM:SS (тип данных TIMESTAMP)
NOW() выдает время самого сервера СУБД
Синтаксис:
SELECT NOW() AS текущее_время;

44. Инструмент_38: Поиск просроченного, [NOW () в WHERE]
Функция NOW() располагается в блоке WHERE для выявления «просрочки»
Пример и синтаксис: 
WHERE plan_date < NOW() AND fact_date IS NULL;
/ Мгновенный вылет всех сорванных дедлайнов на текущую секунду.
 
Глава 13. БЛОК SELECT: ИЗВЛЕЧЕНИЕ ДЕТАЛЕЙ

44. Инструмент_38: Хирургический резак дат — [EXTRACT() в SELECT]
Функция EXTRACT располагается в блоке SELECT. Она заходит в ячейку с полной датой, забирает оттуда чистую изолированную цифру: только ГОД [YEAR], только МЕСЯЦ [MONTH] или только ДЕНЬ [DAY], и выводит её в отдельную колонку в виде обычного целого числа, не забываем про AS.
Синтаксис 
SELECT EXTRACT(что_вырезаем FROM колонка_с_датой)
1.	Вырезаем год: EXTRACT(YEAR FROM s_date) – вернет число 2026.
2.	Вырезаем месяц: EXTRACT(MONTH FROM s_date) – вернет число 5.
3.	Вырезаем день: EXTRACT(DAY FROM s_date) – вернет число 17.
Если требуется отфильтровать отчет по конкретному месяцу, мы обязаны писать функцию целиком: WHERE EXTRACT(MONTH FROM date_fact) = 5. Иначе будет ошибка (WHERE выполняется до того, как SELECT присвоит псевдоним).

45. Инструмент_39: Сжатие по периодам — [EXTRACT() в GROUP BY]
Самый мощный аналитический маневр. Когда мы вырезали из дат только номера месяцев, все дни превратились в одинаковую цифру месяца. Теперь GROUP BY видит эти урезанные месяца от дат как одинаковые строки и схлопывает их в один узел.
SELECT 
    EXTRACT(MONTH FROM date_fact) AS номер_месяца, 
    SUM(quantity) AS общий_объем
FROM supply
GROUP BY 1;

В случае существования в отчете нескольких лет, чтобы года не перемешались в процессе группировок, мы режем дату сразу по двум осям — и по Году, и по Месяцу одновременно:
SELECT 
    EXTRACT(YEAR FROM date_fact) AS год,
    EXTRACT(MONTH FROM date_fact) AS месяц, 
    SUM(quantity) AS объем
FROM supply
GROUP BY 1, 2; 
-- Сгруппирует по уникальным парам (2025-5, 2026-5)
Глава 14. БЛОК SELECT: ОБНУЛЕНИЕ ЧАСОВЫХ ПОЯСОВ

46. Инструмент_40: Тотальное обнуление времени — [DATE_TRUNC() в SELECT]
Главная роль DATE_TRUNC (Date Truncate — обрезка даты) — это принудительное выжигание времени (часов, минут, секунд) из монолитного формата TIMESTAMP.
По умолчанию базы данных сохраняют дату вместе со временем (2026-05-17 14:32:05). Из-за разницы в секундах GROUP BY никогда не сможет объединить эти строки. 
DATE_TRUNC — это топор. Указывая границу (например, день 'day'), система намертво обнуляет всё, что находится правее этой границы, превращая часы и минуты в нули. На выходе остается полноценный календарный день.
Синтаксис:
DATE_TRUNC('до_какого_уровня_режем', колонка_с_датой)
На выходе DATE_TRUNC оставляет тип данных TIMESTAMP

Три режима физической обрезки (Наглядный пример):
Исходное грязное значение: 2026-05-17 14:32:05 (Воскресенье)
1.	Обрезка до Дня ('day'): DATE_TRUNC('day', s_date)
o	Результат: 2026-05-17 00:00:00. (Время выжжено в полночь. Идеально для ежедневных сводок).
2.	Обрезка до Месяца ('month'): DATE_TRUNC('month', s_date)
o	Результат: 2026-05-01 00:00:00. (Время выжжено, а день сброшен на первое число. Удобно для ежемесячных отчетов).
3.	Обрезка до Года ('year'): DATE_TRUNC('year', s_date)
o	Результат: 2026-01-01 00:00:00. (Остался только чистый старт года).
 
47. Инструмент_41: Схлопывание по дням — [DATE_TRUNC() и GROUP BY]
Применяется, когда в базе лежат тысячи операций, зафиксированных с точностью до секунды. Чтобы собрать их в аккуратный посуточный отчет, мы сначала обнуляем время до уровня дня, а затем группируем.
SELECT 
    DATE_TRUNC('day', date_fact) AS чистый_день,
    SUM(quantity) AS объем_за_сутки
FROM supply
GROUP BY 1
Результат: Все операции, выполненные 17 мая в 10:00, в 14:32 и в 23:59, получат одинаковый маркер 2026-05-17 00:00:00 и схлопнутся в одну единственную строку за эти сутки.

ВАЖНО — функция DATE_TRUNC является «родной» для СУБД PostgreSQL. Если в будущем придется работать на серверах MySQL или SQL Server (MS SQL), там этого «топора» нет, а выжигание времени до уровня дня выполняется через другие инструменты (например, CAST(col AS DATE) в SQL Server или DATE(col) в MySQL). 

Всегда помни, на каком движке СУБД развернуто твое поле боя.!

 
Глава 15. ФИНАЛ: ОКОННЫЕ ФУНКЦИИ OVER (PARTITION BY ...)

48. Инструмент_42: Виртуальный GROUP BY – [OVER (PARTITION BY)]
OVER (PARTITION BY) позволяет выполнять агрегатные функции в виртуальном окне, физически не группирую данные в таблице и не меняя физический вид отчета, оставляя все детали.
Данная функция буквально, в виртуальном пространстве, выполняет агрегацию и возвращает результат агрегации каждому участнику каждой существующей группы. 
Синтаксис:
АГРЕГАТ() OVER (PARTITION BY колонка_разбивки) AS имя

Применяя оконные функции, в выводимом отчете мы можем видеть:
Название материала, объем точечной поставки и весь объем данного материала – универсальный пример-ключ.

Универсальный пример - синтаксис
SELECT 
    object_name,
    material_name,
    quantity, 
    SUM(quantity) OVER (PARTITION BY object_name) AS всего_по_объекту
FROM supply;

	Как этот приказ выполняет движок системы:
1.	Система выводит детальные строки: Объект_1 | Арматура | 5 тонн.
2.	Встречая OVER (PARTITION BY object_name), система мгновенно создает «виртуальное окно» для каждой группы (из указанной колонки в скобках partition by COL), суммирует (или иная агрегация) весь его вес (например, 100 тонн) и приклеивает эту сотню к каждой строке этого объекта.
3.	Строки не уничтожаются. Мы видим и точечный объем и объем всей соответствующей группы.

 
Глава 15.1. БЛОК WHERE: ОБХОД ЗАПРЕТА ОКОННЫХ ФУНКЦИЙ

Логика: Система наотрез запрещает писать оконные функции в блоке WHERE. Приказ типа WHERE SUM(quantity) OVER(...) > 100 вызовет мгновенный отказ сервера.

Почему? Потому что WHERE выполняется под капотом на шаге №2, а оконные функции рождаются только на шаге №5 в блоке SELECT. Система физически еще не знает результатов окна, когда пытается отсеять строки.

Решение (Маневр Матрешки): Мы принудительно заставляем виртуальное окно родиться раньше. Мы прячем наш запрос с оконной функцией во FROM, превращая его в чистую производную таблицу. Для внешнего (главного) запроса результат окна становится обычной, готовой колонкой, которую блок WHERE щелкает как орех.

Архитектурный чертеж (Синтаксис обхода)
Задача: Оставить в отчете только те объекты ПТО, где суммарный объем поставок материала превысил лимит в 100 тонн, но при этом сохранить вывод каждой детальной строки.
SELECT 
    t1.object_name,
    t1.material_name,
    t1.quantity,
    t1.всего_по_объекту
FROM (
    -- ВНУТРЕННИЙ ЗАПРОС (МАТРЕШКА): Здесь рождается окно
    SELECT 
        object_name,
        material_name,
        quantity,
        SUM(quantity) OVER (PARTITION BY object_name) AS всего_по_объекту
    FROM supply
) AS t1
-- ВНЕШНИЙ ФИЛЬТР: Здесь мы обходим запрет системы
WHERE t1.всего_по_объекту > 100;



Детальный разбор работы движка (Пошаговый обход под капотом)
1.	Шаг 1. Сборка внутреннего мира: Сервер заходит во внутренний запрос, считывает таблицу supply и запускает оконную функцию SUM(...) OVER(...).
2.	Шаг 2. Рождение легального имени: Оконный агрегат вычисляется для каждой строки, и ему присваивается жесткое имя всего_по_объекту.
3.	Шаг 3. Материализация острова: Весь этот результат выплескивается наружу в блок FROM главного запроса и запечатывается под позывным AS t1. Теперь это обычная базовая таблица.
4.	Шаг 4. Легальный отсев: Включается внешний блок WHERE. Он смотрит на таблицу t1 и видит в ней готовую статичную колонку всего_по_объекту. Для него это больше не оконная функция, а просто набор чисел. Блок WHERE безболезненно отсекает строки, где значение <= 100.
5.	Шаг 5. Финальный фасад: Главный SELECT на самом верхнем уровне забирает очищенные строки и выводит их на экран.
 
Глава ЭПИЛОГ: ИНСТРУМЕНТЫ ЮВЕЛИРА — СВЕРХТОЧНОЕ ЛИТЬЕ ТИПОВ (:: NUMERIC)
49. Инструмент_43: Быстрый каст точности — :: numeric (PostgreSQL-стандарт)
Логика: Это синтаксический «быстрый нож» (сокращенный оператор), который ставится сразу после имени колонки или числа через двойное двоеточие. 
Он принудительно трансформирует тип данных в NUMERIC (он же DECIMAL).
Главный биологический смысл инструмента — принудительное добавление дробного хвоста к числам прямо в оперативной памяти, чтобы перевести систему из режима «грубого счета» в режим «сверхточного финансового/инженерного литья».
Синтаксис:
SELECT column::numeric(precision, scale) AS имя_колонки
•	:: — Сигнал системе: «Внимание, мгновенный CAST».
•	numeric — Целевой тип данных. Точное число с фиксированной запятой.
•	precision (Точность) — Общее количество цифр в числе с обеих сторон от запятой (весь масштаб числа).
•	scale (Масштаб) — Количество цифр строго после запятой (дробная часть).

Внутренняя физика процесса (Как думает система)
В SQL типы данных делятся на Целочисленные (INT, BIGINT) и Вещественные (NUMERIC, FLOAT).
Когда ты применяешь ::numeric, ты меняешь физику процесса:
SELECT 5::numeric(10,2) / 2; -> Под капотом: 5 превращается в 5.00. Теперь это NUMERIC / INT. Система обязана сохранить точность старшего типа. Результат: 2.5000000... (в зависимости от настроек базы).
Капканы и Брешь-контроль
Защита: Если не уверен в финальном объеме цифр, пиши ::numeric без указания скобок. Система сама выделит динамический максимум памяти под результат, и отчет не упадет.
Защита: Применяй инструмент до совершения математического действия, непосредственно к делителю или делимому: SELECT 5::numeric / 2.
 
Глава 9.1. БЛОК SELECT, ОКЛЕМЕНТЫ — ЖЕСТКИЙ НУМЕРАТОР
27. Инструмент_21: Сквозной счетчик строк — ROW_NUMBER() OVER()
Логика: Это оконная функция, которая генерирует непрерывную последовательность целых чисел (1, 2, 3, 4...) для каждой строки в рамках заданного виртуального «окна». Даже если две строки абсолютно идентичны по своему содержимому, ROW_NUMBER() безжалостно присвоит им разные номера на основе их физического порядка в выводе.
Синтаксис монолитный:
sql
ROW_NUMBER() OVER (
    PARTITION BY колонка_группировки 
    ORDER BY колонка_сортировки ASC/DESC
) AS имя_колонки
Используйте код с осторожностью.
•	ROW_NUMBER() — Сам приказ: «Включить счетчик». Скобки всегда пустые.
•	OVER — Дверь в оконный режим. Сигнал системе: «Не схлопывай строки, работай внутри ряда».
•	PARTITION BY — Стена отсека. Делит всю таблицу на изолированные группы (мини-миры). Счетчик сбрасывается в 1 при переходе в новый отсек.
•	ORDER BY — Колея счетчика. Задает жесткий порядок, в котором строки будут подходить к маркеру нумерации.

2.	Шаг 2 (Построение): Внутри каждого кармана выстроит строки по дате от самой ранней к поздней (ASC).
3.	Шаг 3 (Штамп): Пройдет пешком сверху вниз и проставит номера.

🚨 Капканы и Брешь-контроль
•	Капкан 1: Попытка прямой фильтрации в WHERE.
Ты хочешь вытащить только самые первые поставки и пишешь:
Ошибка: WHERE ROW_NUMBER() OVER (...) = 1.
Запрос упадет намертво. Помни закон Logical Query Processing (Гибрид №5): блок WHERE выполняется раньше, чем блок SELECT. На этапе фильтрации оконных номеров еще физически не существует.
o	Защита (Маневр Матрешки через FROM): Сначала зашиваем окно внутрь производной таблицы, а во внешнем «приоритетном» запросе спокойно режем результат.
sql
SELECT t1.object_id, t1.delivery_date, t1.volume_tons
FROM (
    SELECT object_id, delivery_date, volume_tons,
           ROW_NUMBER() OVER (PARTITION BY object_id ORDER BY delivery_date ASC) AS rn
    FROM supply
) AS t1
WHERE t1.rn = 1; -- Теперь WHERE видит готовую колонку "rn"
Используйте код с осторожностью.
•	Капкан 2: Нестабильная колея (Амбивалентность сортировки).
Если у тебя две поставки по одному объекту прошли в один и тот же день (время не указано), а ты сортируешь только по дате — система выдаст им номера 1 и 2 случайным образом. При каждом перезапуске запроса они могут меняться местами.
o	Защита: Всегда добавляй в ORDER BY уникальный дополнительный критерий для однозначности (например, ORDER BY delivery_date ASC, id ASC).

2. Срез (Эталон для бункера)
•	Суть: Сквозная жесткая нумерация строк (1, 2, 3...) без сжатия таблицы.
•	Движок: PARTITION BY режет на группы, ORDER BY задает колею движения счетчика.
•	Главная роль: Поиск экстремумов (самый первый заказ, самый последний этап, топ-3 дорогих материалов).
•	Закон выживания: Окно пишется только в SELECT. Для фильтрации результата (rn = 1) подзапрос в блоке FROM — обязателен.
 
Глава 9.2. БЛОК SELECT, ОКЛЕМЕНТЫ — ПЛОТНЫЙ РАНЖИРОВЩИК
28. Инструмент_22: Медальный зачет без пропусков — DENSE_RANK() OVER()
Логика: Это оконная функция, которая присваивает ранг (место в рейтинге: 1, 2, 3...) каждой строке внутри заданного «окна» [1]. Если две или более строки имеют абсолютно одинаковые значения в колонке сортировки (ORDER BY), система выдает им одинаковое место. При этом следующая за ними уникальная строка получит строго следующий по порядку номер, без перескоков. Слово DENSE переводится как «плотный» — это значит, что числовая последовательность рангов никогда не разорвется.
Синтаксис монолитный:
sql
DENSE_RANK() OVER (
    PARTITION BY колонка_группировки 
    ORDER BY колонка_ранжирования ASC/DESC
) AS имя_колонки
Используйте код с осторожностью.
•	DENSE_RANK() — Приказ: «Присвоить места по плотной шкале». Скобки всегда пустые [1].
•	PARTITION BY — Стена отсека. Счетчик сбрасывается в 1 при переходе к следующей группе объектов [1].
•	ORDER BY — Весы ранжирования. Колонка, по значению которой система определяет, кто достоин первого места, а кто — второго.

⚙️ Внутренняя физика процесса (Разница между RANK и DENSE_RANK)
Чтобы ядро приняло этот инструмент без сбоев, нужно увидеть его отличие от классического RANK() и ROW_NUMBER().
Представь соревнование сварщиков на объекте ПТО. Трое из них сдали швы идеального качества (оценка 100), а четвертый сдал чуть хуже (оценка 95). Смотрим, как сработают три разных счетчика:
1.	ROW_NUMBER() выдаст: 1, 2, 3, 4. (Жесткий конвейер. Ему плевать на равенство, он принудительно разделит одинаковых сварщиков).
2.	RANK() выдаст: 1, 1, 1, 4. (Спортивный зачет с дырой. Выдал трем лидерам первое место, но «сожрал» 2-е и 3-е места. Следующий получил сразу 4-е).
3.	DENSE_RANK() выдаст: 1, 1, 1, 2. (Плотный зачет. Выдал трем лидерам первое место, но сохранил непрерывность цепочки. Сварщик с оценкой 95 получает честное 2-е место).

🚨 Капканы и Брешь-контроль
•	Капкан 1: Коварство сортировки по умолчанию (ASC).
Если ты ранжируешь материалы по цене, но забыл указать DESC, система начнет выдавать 1-е место самому дешевому материалу, так как по умолчанию сортирует от меньшего к большему.
o	Защита: Если цель — найти ТОП самых дорогих, масштабных или приоритетных позиций, всегда явно пиши DESC внутри окна: ORDER BY price DESC.
•	Капкан 2: Тот же закон WHERE.
Как и в случае с ROW_NUMBER(), ты не имеешь права использовать DENSE_RANK() напрямую в блоке WHERE. Чтобы отсечь только топ-1 или топ-2 позиций (включая всех, кто разделил эти места), обязателен маневр «Матрешки» (производной таблицы во FROM).
sql
SELECT t1.material_name, t1.price, t1.price_rank
FROM (
    SELECT material_name, price,
           DENSE_RANK() OVER (ORDER BY price DESC) AS price_rank
    FROM supply_materials
) AS t1
WHERE t1.price_rank <= 2; -- Выведет все материалы на 1-м и 2-м местах, со всеми тёзками по цене
Используйте код с осторожностью.

2. Срез (Эталон для бункера)
•	Суть: Присвоение мест (рангов) с учетом равенства значений и без числовых пропусков (1, 1, 2, 3...).
•	Отличие: В отличие от RANK, не оставляет дыр в нумерации. В отличие от ROW_NUMBER, выдает одинаковые номера для равных значений.
•	Главная роль: Вычисление плотных топ-листов (найти ТОП-3 самых дорогих марок бетона, даже если на первом месте стоят сразу пять поставщиков с одинаковой ценой).
 
Глава 9.3. БЛОК SELECT, ОКЛЕМЕНТЫ — СДВИГАТЕЛИ ПОТОКА
29. Инструмент_23: Шаг назад (Машина времени) — LAG() OVER()
Логика: Эта оконная функция возвращает значение из указанной колонки той строки, которая находится выше (раньше) текущей строки на заданный шаг внутри виртуального окна. Она позволяет сравнить текущую ячейку с её прошлым состоянием.
Синтаксис монолитный:
sql
LAG(целевая_колонка, шаг_сдвига, заглушка_NULL) OVER (
    PARTITION BY колонка_группировки 
    ORDER BY колонка_сортировки ASC/DESC
) AS имя_колонки
Используйте код с осторожностью.
•	целевая_колонка — Откуда воровать данные сзади (например, volume_tons).
•	шаг_сдвига — На сколько строк назад отступить. Если не указать, по умолчанию равен 1 (предыдущая строка).
•	заглушка_NULL — Опциональный параметр. Что подставить, если строки сзади нет (для самой первой строки в окне). Если не указать, система вернет NULL.

Логика: Полный зеркальный аналог функции LAG. Она заглядывает вперед (вниз) по потоку и вытягивает значение из строки, которая будет обработана позже.
Синтаксис монолитный:
sql
LEAD(целевая_колонка, шаг_сдвига, заглушка_NULL) OVER (
    PARTITION BY  колонка_группировки 
    ORDER BY колонка_сортировки ASC/DESC
) AS имя_колонки
Используйте код с осторожностью.

o	Защита: Это не баг, это фича. Главное — помни об этом и используй встроенный третий параметр (заглушку) или внешний COALESCE, чтобы расчеты дельты (Текущий - Предыдущий) не ломались об этот закон.

•	Глава 3.6. БЛОК SELECT, СВЕРХТОЧНОЕ ЛИТЬЕ — DECIMAL vs NUMERIC
•	Прямой ответ в одно предложение: В современных СУБД (PostgreSQL, SQL Server, MySQL) DECIMAL и NUMERIC функционально идентичны, используют одинаковый объем памяти и являются синонимами. Однако на уровне официального стандарта ANSI/ISO SQL между ними заложено фундаментальное различие в жесткости удержания масштаба цифр. [1, 2, 3]
•	________________________________________
•	⚙️ Глубинная дешифровка (Разница по стандарту ANSI SQL)
•	Когда ты объявляешь параметры (p, s), где p (Precision) — общее число цифр, а s (Scale) — количество знаков после запятой: [1]
•	NUMERIC(p, s) — Дисциплинированный диктатор (Точно сколько просили).
По стандарту этот тип обязан хранить ровно указанную точность p. Если ты задал NUMERIC(5,2), система выделит физическую память строго под 5 цифр. Никакой самодеятельности под капотом.
•	DECIMAL(p, s) — Лояльный инженер (Не меньше, чем просили).
Стандарт разрешает системе вольность: база данных обязана обеспечить точность не меньше указанной p, но если движку СУБД технически проще выделить ячейку памяти с запасом (например, под 6 или 8 цифр из-за особенностей разрядности процессора) — она имеет на это право. [1, 2, 3, 4, 5]
•	________________________________________
•	🚨 Капкан реальной практики: Иллюзия равенства в СУБД
•	Большинство современных баз данных (включая PostgreSQL) решили не плодить разные сущности. Они просто сделали DECIMAL физическим алиасом (псевдонимом) для NUMERIC. Под капотом они работают по одному коду. [1, 2]
•	Но здесь тебя поджидает Капкан константной жесткости (Сбой Foreign Key):
Если ты проектируешь архитектуру базы данных (мосты между таблицами) и связываешь две таблицы по ключевым колонкам (например, total_cost в Сметах и paid_cost в Платежах) — они обязаны быть одного типа. [1, 2]
•	Ошибка: Объявить в левой таблице NUMERIC(10,2), а в правой таблице-партнере DECIMAL(10,2). Некоторые СУБД (например, MS SQL Server) заблокируют построение связи FOREIGN KEY с ошибкой несоответствия типов данных, несмотря на их внутреннюю синонимичность. [1, 2]
•	________________________________________
•	2. Срез (Эталон для бункера)
•	В реальности СУБД: Полные синонимы. Память, скорость работы и результат вычислений совпадают на 100%.
•	В теории стандарта: NUMERIC — строго p цифр (фиксировано). DECIMAL — не менее p цифр (допускается внутренний запас памяти системы).
•	Закон Дисциплины: Выбери один тип для всего проекта и жестко придерживайся его. Для быстрых кастов в PostgreSQL используй ::numeric — это стандарт де-факто. [1, 2, 3, 4, 5]
 
•	Глава 3.7. БЛОК SELECT, ТЕКСТОВАЯ МАТЕРИЯ — VARCHAR
•	31. Инструмент_25: Динамический контейнер текста — VARCHAR(N)
•	Логика: Название расшифровывается как Variable Character (символы переменной длины). Это тип данных, предназначенный для хранения строк (букв, цифр, знаков препинания, пробелов). Главный биологический смысл инструмента — экономия памяти сервера. Он выделяет ровно столько байт, сколько символов реально вбито в ячейку, ограничивая лишь максимальный потолок.
•	Синтаксис: VARCHAR(N)
•	N — жесткий потолок. Максимально допустимое количество символов, которое система разрешит записать в эту ячейку.
•	Пример: VARCHAR(100) — контейнер вместит любое слово от 0 до 100 символов, но не пропустит строку в 101 символ.
•	________________________________________
•	⚙️ Внутренняя физика процесса: Сравнение VARCHAR vs CHAR
•	Чтобы ядро приняло инструмент без сбоев, нужно сравнить его с его предшественником — типом CHAR(N) (фиксированная длина).
•	Представь, что у нас есть коробка на 10 ячеек (N=10), а мы кладем туда короткое слово 'ПТО' (3 символа).
•	Тип CHAR(10) — Жесткий монолит:
Система положит слово 'ПТО', а оставшиеся 7 ячеек принудительно забьет пустыми пробелами под капотом. На выходе в базе будет лежать 'ПТО '. Ты будешь платить за хранение пустоты на жестком диске.
•	Тип VARCHAR(10) — Живая резина:
Система положит слово 'ПТО', увидит, что занято всего 3 символа, и сожмет физический размер ячейки до 3 символов (+ 1-2 байта на технический маркер длины). Пустые пробелы справа не создаются

 
нужен полный и детальный разбор CASE в SUM, как работает
Конструкция SUM(CASE ... END) в SQL используется для выборочного (условного) суммирования. Она позволяет сложить значения из строк, которые соответствуют конкретному условию, игнорируя остальные.
________________________________________
💡 Главный секрет: как они работают вместе
По отдельности эти операторы работают стандартно: CASE проверяет строки по очереди, а SUM складывает результат. Когда вы их объединяете, СУБД выполняет запрос в три шага:
1.	Сканирование: СУБД идет по таблице строка за строкой.
2.	Фильтрация (CASE): Для каждой строки проверяется условие. Если оно верно (TRUE), возвращается указанное число или колонка. Если неверно (FALSE), возвращается 0 (или NULL).
3.	Агрегация (SUM): Все полученные из CASE значения складываются в одну итоговую сумму.
________________________________________
🛠️ Базовый синтаксис
sql
SELECT 
    SUM(CASE 
            WHEN условие THEN колонка_или_число
            ELSE 0 
        END) AS название_поля
FROM таблица;
Используйте код с осторожностью.
Важно: Всегда используйте ELSE 0. Если его опустить, не подошедшие под условие строки вернут NULL. Суммирование с NULL не сломает расчет (SUM игнорирует NULL), но явный ELSE 0 делает код понятнее и безопаснее.
________________________________________
📖 Пошаговый разбор на примере
Представьте таблицу продаж orders:
id	product	category	amount
1	iPhone	Electronics	1000
2	Jeans	Apparel	100
3	Laptop	Electronics	2000
4	T-Shirt	Apparel	50
Мы хотим посчитать сумму продаж только для категории 'Electronics'.
Запрос:
sql
SELECT 
    SUM(CASE 
            WHEN category = 'Electronics' THEN amount 
            ELSE 0 
        END) AS electronics_total
FROM orders;
Используйте код с осторожностью.
Как это видит база данных изнутри:
•	Строка 1: Категория 'Electronics'? Да. Возвращаем amount -> 1000.
•	Строка 2: Категория 'Electronics'? Нет. Срабатывает ELSE -> 0.
•	Строка 3: Категория 'Electronics'? Да. Возвращаем amount -> 2000.
•	Строка 4: Категория 'Electronics'? Нет. Срабатывает ELSE -> 0.
Финальный шаг (SUM): 1000 + 0 + 2000 + 0 = 3000.
Результат: 3000.

⚡ Основные сценарии применения
1. Подсчет количества строк по условию (Аналог COUNT)
Вместо суммирования бизнес-показателей мы пишем THEN 1. Каждая подошедшая строка прибавит к сумме единицу.
sql
-- Считаем отдельно количество дорогих и дешевых покупок
SELECT 
    SUM(CASE WHEN amount >= 1000 THEN 1 ELSE 0 END) AS premium_orders_count,
    SUM(CASE WHEN amount < 1000 THEN 1 ELSE 0 END) AS cheap_orders_count    
FROM orders;
Используйте код с осторожностью.
2. Разворот таблицы (Pivot)
Это самый частый кейс. Вместо вывода категорий в разные строки через GROUP BY, мы выводим их в отдельные столбцы.
sql
-- Было: строки. Стало: плоская таблица с колонками под каждую категорию
SELECT 
    SUM(CASE WHEN category = 'Electronics' THEN amount ELSE 0 END) AS electronics_sales,
    SUM(CASE WHEN category = 'Apparel' THEN amount ELSE 0 END) AS apparel_sales
FROM orders;
Используйте код с осторожностью.
3. Суммирование с несколькими условиями (AND / OR)
Внутри WHEN можно собирать сложные логические цепочки.
sql
SELECT 
    SUM(CASE 
            WHEN category = 'Electronics' AND amount > 1500 THEN amount 
            ELSE 0 
        END) AS high_end_electronics_sales
FROM orders;
Используйте код с осторожностью.

⚠️ Частые ошибки новичков
•	Забытый END: CASE обязательно должен закрываться ключевым словом END. Без него возникнет синтаксическая ошибка.
•	Путаница с COUNT: Если написать COUNT(CASE WHEN условие THEN amount ELSE 0 END), то база данных посчитает все строки. Функция COUNT считает количество любых значений, отличных от NULL, а 0 — это значение. Для COUNT нужно использовать ELSE NULL (или просто опускать ELSE). Для SUM безопаснее ELSE 0.
•	Суммирование внутри CASE: Запись вида CASE WHEN ... THEN SUM(amount) END ошибочна. Сначала должна идти групповая функция SUM, а внутри нее — построчный фильтр CASE.
 
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
Эта конструкция определяет окно кадра (window frame) в оконных функциях SQL. Она управляет тем, какие именно строки из текущей секции (партиции) будут попадать в расчет для каждой конкретной строки.
Если перевести фразу дословно, она означает: «От самой первой строки секции до текущей строки включительно».
________________________________________
⚙️ Разбор каждого слова (Синтаксис)
•	ROWS: Указывает, что границы окна определяются физическим количеством строк (строка за строкой), а не значениями в них.
•	BETWEEN ... AND ...: Задает начальную и конечную точку диапазона.
•	UNBOUNDED PRECEDING: Начальная точка. Означает «без ограничений сзади» — то есть расчет всегда начинается с самой первой строки в текущей партиции.
•	CURRENT ROW: Конечная точка. Расчет останавливается на текущей строке, которую сейчас обрабатывает база данных.
________________________________________
📈 Главный кейс: Нарастающий итог (Running Total)
Эта конструкция — стандарт индустрии для подсчета кумулятивной суммы, скользящего среднего или прогрессивного количества.
Пример на таблице продаж daily_sales:
Date (Дата)	Revenue (Выручка)
2026-01-01	100
2026-01-02	150
2026-01-03	200
SQL-запрос:
sql
SELECT 
    Date, 
    Revenue,
    SUM(Revenue) OVER(
        ORDER BY Date 
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS Running_Total
FROM daily_sales;
Используйте код с осторожностью.
Как это работает пошагово для каждой строки:
1.	На строке 1 (01-01): Окно включает строки с первой по текущую (только строку 1).
o	Что складываем: 100
o	Результат: 100
2.	На строке 2 (01-02): Окно расширяется со второй строки назад до первой (строки 1 и 2).
o	What складываем: 100 + 150
o	Результат: 250
3.	На строке 3 (01-03): Окно включает все строки от начала до третьей (строки 1, 2 и 3).
o	Что складываем: 100 + 150 + 200
o	Результат: 450
Итоговая таблица:
Date	Revenue	Running_Total
2026-01-01	100	100
2026-01-02	150	250
2026-01-03	200	450
________________________________________
❗ Ключевое отличие: ROWS против RANGE (Опасная ловушка)
Если вы просто напишете ORDER BY Date и не укажете рамки окна, база данных по умолчанию применит скрытое правило: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.
Разница между ROWS и RANGE критична, если в таблице есть одинаковые значения в колонке сортировки (ORDER BY).
•	ROWS жестко смотрит на физические строки. Ему все равно на дубликаты. Он будет прибавлять строки строго по одной.
•	RANGE смотрит на значения. Если у двух строк одинаковая дата, RANGE объединит их и прибавит их сумму одновременно для обеих строк.
Пример с дубликатами дат:
Если 2 января было две продажи по 150:
Date	Revenue	Результат с ROWS	Результат с RANGE (По умолчанию)
2026-01-01	100	100	100
2026-01-02	150	250 (100+150)	400 (Расчет сразу за оба дня: 100+150+150)
2026-01-02	150	400 (100+150+150)	400
Правило: Для классического, плавного нарастающего итога всегда явно пишите ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. Это застрахует от искажения данных и во многих СУБД (например, PostgreSQL) отработает быстрее, так как базе не нужно искать дубликаты значений.
________________________________________
🧩 Работа с секциями (PARTITION BY)
Если добавить PARTITION BY, то при переходе к новой группе строк окно «сбрасывается», и расчет кумулятивной суммы начинается заново.
sql
SELECT 
    Employee_ID, Date, Bonus,
    SUM(Bonus) OVER(
        PARTITION BY Employee_ID -- Сброс для каждого сотрудника
        ORDER BY Date 
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS Cumulative_Bonus
FROM Salaries;
Используйте код с осторожностью.

Сценарий 4: Технический ID (Когда порядок вообще не важен)
Самый частый технический трюк. Вам абсолютно всё равно, в каком порядке строки сложатся, вам просто нужно, чтобы они считались строго по очереди, а не одной кучей.
•	Бизнес-задача: Сделать плавный нарастающий итог по заказам, но дат и времени в таблице нет.
•	Что ставим в ORDER BY: Первичный ключ таблицы, уникальный номер строки (id).
•	Запрос: SUM(amount) OVER(ORDER BY id)
•	Как идет очередь: Строки просто выстроятся по своему системному номеру: 1, 2, 3, 4... Это гарантирует, что калькулятор прибавит их по одной и результат всегда будет предсказуемым.

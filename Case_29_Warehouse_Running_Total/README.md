# Кейс №29: Оконные развилки CASE WHEN и накопительный итог баланса склада

## Блок: Промышленная аналитика / Логистика и складской учет
**Заказчик:** Начальник отдела логистики / Коммерческий директор.

### Бизнес-задача (ТЗ):
> *"Для автоматизации складского учета и предотвращения разрывов в цепочках поставок материально-технических ресурсов необходимо разработать аналитический инструмент операционного контроля. Требуется сформировать ведомость на основе лога складских операций с выводом идентификаторов логов и нормализованных имен операторов. В смежном третьем столбце необходимо реализовать скользящий расчет текущего баланса остатков в разрезе каждого оператора. Критическое условие: алгоритм должен на лету изменять математический знак операции (прибавлять объемы при операциях 'ПРИХОД' и вычитать со знаком минус при операциях 'ВОЗВРАТ'). Расчет должен нарастать пошагово без группового сжатия строк. Результаты упорядочить по алфавиту и хронологии."*

---

## Исходная структура СУБД (Таблица `warehouse_logs`):
*   `log_id` (INT) — Уникальный номер складского лога / накладной (Первичный ключ).
*   `operator_name` (VARCHAR) — ФИО оператора склада, зафиксировавшего операцию (содержит грязный ввод).
*   `operation_type` (VARCHAR) — Тип проводимой складской операции (например, 'ПРИХОД', 'ВОЗВРАТ').
*   `quantity` (INT) — Количество материальных ценностей в штуках/тоннах (содержит пропуски `NULL`).

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического конвейера был применен сложнейший паттерн Enterprise-разработки — интеграция условного ветвления `CASE WHEN` непосредственно внутрь скользящего оконного агрегатора `SUM() OVER()`. Это позволяет СУБД осуществлять многофакторный учет и кумулятивное сложение разнонаправленных коммерческих потоков за один физический проход вычислительного ядра по таблице, минуя тяжелые процедуры сопряжения.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Первичная обработка на верхнем уровне (`SELECT`):** Сервер СУБД извлекает исходные поля, попутно вычищая имена операторов от скрытого ручного пробельного шума и регистровых ошибок комбо-резцами `TRIM(LOWER())`.
2. **Внедрение знакового переключателя (`CASE WHEN внутри SUM`):** Движок СУБД бежит по строкам лога и на лету переключает математический знак транзитной метрики: при совпадении со значением 'ПРИХОД' поле `quantity` берется в чистый плюс, при значении 'ВОЗВРАТ' перед функцией `COALESCE` принудительно выбивается математический токен минуса (`-`), трансформируя операцию в вычитание. Прочие технические строки зануляются через `ELSE 0`.
3. **Секционирование и хронологическая привязка:** Конструкция `PARTITION BY` изолирует операционные контуры в герметичные независимые расчетные окна по каждому оператору. Оператор `ORDER BY log_id` выстраивает накладные в строгую временную рельсовую цепочку от старых к новым.
4. **Скользящий накопительный подсчет:** Оконная рамка `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` приказывает серверу запускать ползущий кумулятивный счетчик (в Python Pandas логика зеркально реализована через знаковую маску `np.select` и векторный групповой метод `.cumsum()`). Движок ювелирно прибавляет к сумме накопленного прошлого результата дельту текущей строки-остановки, формируя стерильное поле баланса без групповых скачков на совпадающих временных метках.
5. **Выравнивание витрины:** Финальный глобальный оператор `ORDER BY 2, 1` структурирует итоговый лог перед руководством по алфавиту операторов и номерам накладных.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и верификации сложного кумулятивного конвейера склада были зафиксированы и устранены следующие технические риски и ошибки сбоя:

*   **Риск №1 (Критическая синтаксическая отсечка — потеря токена CASE):** Фатальное падение компилятора базы данных на этапе парсинга из-за пропущенного доминантного слова-оператора `CASE` перед условием (`SUM( WHEN operation_type = ... )`). Сервер СУБД не мог идентифицировать синтаксическое семейство команды и блокировал выполнение всего Highload-сервера.
    *   *Метод устранения:* Условный переключатель восстановлен в соответствии с каноническим стандартом: `SUM(CASE WHEN ... )`.
*   **Риск №2 (Ловушка потерянной скобки закрытия оператора и отвал окна OVER):** Ошибка разбора синтаксиса из-за дублирования закрывающих скобок непосредственно после тела условной конструкции (`ELSE 0 END) ) OVER`). Функция `SUM` принудительно закрывалась дважды, из-за чего последующий оконный блок `OVER` ложно отваливался от математического контура.
    *   *Метод устранения:* Контур скобочных границ отрегулирован, одиночная скобка закрывает агрегат строго после объявления границ оконной рамки.
*   **Риск №3 (Механический пропуск буквы в имени физического поля сортировки):** Падение сервера на этапе вычисления окна из-за пропущенного символа `d` в названии ключевой колонки (`ORDER BY log_i`). База PostgreSQL выдавала жесткий отказ: «Колонка log_i не существует».
    *   *Метод устранения:* Проведен точечный аудит структуры, целевое физическое поле `log_id` возвращено в контур сортировки окна.
*   **Риск №4 (Техническая помарка в финальном операторе сортировки витрины):** Ошибка парсинга в самом конце скрипта из-за пропущенного символа `R` в команде выравнивания (`ODER BY 2, 1`), вызванная высокой скоростью набора текста на передовой конторы ПТО.
    *   *Метод устранения:* Орфография команды исправлена до промышленного стандарта `ORDER BY`.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная кумулятивная витрина логистики склада под управлением условного окна интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного контроля движения ТМЦ.

### Структура дашборда:
1. **График нарастающего операционного баланса (Warehouse Stock Trend):** Накопительная линейная диаграмма, отражающая скользящие остатки на складе по каждому оператору, позволяющая руководству визуально контролировать динамику запасов и ловить точки кассовых и товарных разрывов.
2. **Индикатор суточной оборачиваемости (KPI Card):** Крупный цифровой виджет, отображающий чистый объем прихода готовой продукции за текущие сутки для экспресс-оценки загрузки складских терминалов.
3. **Реестр критических отклонений возвратов:** Таблица, автоматически аккумулирующая операции типа 'ВОЗВРАТ' с максимальным объемом списаний для проведения внутреннего претензионного аудита.

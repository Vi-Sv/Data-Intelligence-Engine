# Кейс №5: Сегментация производственных линий по критическому порогу брака

## Блок: Промышленная аналитика / Управление качеством (ОТК)
**Заказчик:** Директор по производству.

### Бизнес-задача (ТЗ):
> *"Необходимо провести анализ производственных логов департамента контроля качества (ОТК) и локализовать технологические линии с критическими показателями брака. Требуется рассчитать суммарный объем зафиксированных дефектов в разрезе производственных линий, учитывая в расчете исключительно те позиции, которые верифицированы контролером ОТК статусом 'брак'. На финальную витрину необходимо вывести только те производственные узлы, где совокупный объем брака превысил аварийный порог в 15 000 единиц."*

---

## Исходная структура СУБД (Таблица `raw_manufacture`):
*   `item_id` (INT) — Уникальный номер партии (Первичный ключ).
*   `line_id` (INT) — Идентификатор производственной линии (содержит скрытые концевые пробелы).
*   `inspector_comment` (VARCHAR) — Текстовый комментарий контролера ОТК (содержит хаотичный регистр ввода и невидимые символы).
*   `defect_count` (INT) — Объем зафиксированных дефектов в штуках (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Технический аудит):

В ходе предварительного аудита сырого массива данных `raw_manufacture` были локализованы три ключевые уязвимости:
1. Неконсистентность текстового поля комментариев ОТК вследствие ручного ввода данных операторами (различный регистр, наличие начальных и концевых пробелов);
2. Наличие скрытых концевых пробелов в идентификаторах производственных линий, что при стандартной агрегации приводит к искусственному дроблению строк и искажению отчетности;
3. Наличие пропусков типа `NULL` в ячейках подсчета брака.

Для формирования верифицированной витрины реализован следующий аналитический алгоритм:
1. **Первичная фильтрация потока данных:** На уровне строк СУБД с помощью конструкции `WHERE TRIM(LOWER(inspector_comment)) = 'брак'` выполняется очистка и приведение текстовых маркеров к единому регистру. Строки, не содержащие подтвержденный статус дефекта, отсекаются до этапа агрегации, что снижает нагрузку на вычислительный движок.
2. **Нормализация идентификаторов и валидация метрик:** Группировка данных осуществляется по нормализованному полю `TRIM(line_id)` (в Python Pandas: `.str.strip()`), что устраняет проблему скрытых пробелов в номерах линий. Внутри агрегатора `SUM` функция `COALESCE` преобразует пропуски `NULL` в числовое значение `0`, предотвращая логические сбои при математическом суммировании.
3. **Пост-фильтрация агрегированных показателей:** С помощью конструкции `HAVING` (в Python Pandas: последующая фильтрация сгруппированного датафрейма) выполняется финальная сепарация данных. Из выборки исключаются все производственные линии, суммарный показатель брака которых не достиг установленного аварийного порога в 15 000 единиц.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование дашборда ОТК)
Очищенная производственная витрина интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного мониторинга внештатных ситуаций на производстве.

### Архитектура и состав дашборда:
1. **Экстренный индикатор аварийных зон (KPI Card):** Интерактивная таблица, выводящая связку «Номер линии -> Суммарный объем брака» с автоматическим условным форматированием (подсветка линий с максимальным объемом дефектов красным цветом). Позволяет Директору по производству мгновенно локализовать очаг ЧП.
2. **Диаграмма распределения дефектов (Pie / Donut Chart):** Круговая диаграмма, отображающая долевой вклад каждой аварийной линии в общий объем брака предприятия. Используется для оценки масштаба сбоев.
3. **Модуль исторического тренда (Bar Chart):** Столбчатая диаграмма динамики брака, позволяющая сопоставить текущие критические показатели с историческими данными для выявления системных сбоев в оборудовании конкретной линии.
